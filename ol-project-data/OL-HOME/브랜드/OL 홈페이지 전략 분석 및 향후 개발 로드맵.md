# OL 홈페이지 전략 분석 및 향후 개발 로드맵

**기준일**: 2026-06-01
**분석 관점**: 웹 개발 전문가 + 불교 콘텐츠 기획자
**대상**: OL HOME (ol-home 저장소, olbit.org)

---

## 1. 현재 상태 진단

### 1.1 잘 되어 있는 것

OL HOME은 출범 약 2주 만에 놀라울 정도로 건강한 기반을 갖추었다.

**아키텍처**
- Astro SSG + GitHub Pages 조합은 OL의 "로컬 우선, 서버 없음" 철학에 정확히 부합한다.
- 콘텐츠 컬렉션(content.config.ts)이 7개 컬렉션으로 체계화되어 있고, 스키마가 견고하다.
- 도구(ATLAS)와 콘텐츠(BOOK)의 분리 원칙이 코드 구조에 잘 반영되어 있다.

**콘텐츠 구조**
- ATLAS → BOOK → WORKS → DESIGN → BLOG의 파이프라인이 명확하다.
- Entity/Ontology 시스템이 초기 단계부터 설계되어 있어 장기적 확장 기반이 있다.
- 검색(Pagefind) 통합이 이미 구현되어 있다.

**브랜드**
- 선언문, 브랜드 정의서, 결정 카드 등 철학적 기반 문서가 이례적으로 탄탄하다.
- "무엇을 하지 않을 것인가"를 명확히 선언한 것은 프로젝트 정체성의 강력한 방어선이다.

**개발 프로세스**
- `.ol-ref/` 디렉토리를 통한 내부 문서화 습관이 좋다.
- 매뉴얼이 실무적이고 구체적이다.

### 1.2 부족한 것 — 기능 관점

| 영역 | 현재 상태 | 부족한 점 |
|------|-----------|-----------|
| SEO / 소셜 공유 | meta description만 존재 | Open Graph, Twitter Card, 구조화 데이터 없음 |
| 접근성(a11y) | 기본 시맨틱 마크업 | skip-nav, aria-label 불완전, 키보드 탐색 미검증 |
| 성능 | 이미지 최적화 없음 | Astro Image 미사용, WebP 변환 없음, CLS 관리 없음 |
| RSS / 피드 | 없음 | 블로그 구독 수단 없음 |
| i18n | 한국어 단일 | 영문 페이지 없음 (국제 불교 커뮤니티 접근 불가) |
| 오프라인 | 없음 | PWA / Service Worker 미적용 |
| 사이트맵 | 없음 | Google 인덱싱 비효율 |
| 에러 페이지 | Astro 기본 404 | OL 브랜드에 맞는 커스텀 404 없음 |
| OLStats | 주석 처리됨 | 홈 화면 통계가 비활성 |
| Entity 페이지 | 기본 구조만 | 관계 그래프 시각화가 placeholder |

### 1.3 부족한 것 — 콘텐츠 관점

| 영역 | 현재 상태 | 부족한 점 |
|------|-----------|-----------|
| BOOK | 2권 (buddha-story + placeholder) | 완성된 콘텐츠 1권 뿐 |
| WORKS | 5편 | 시리즈 완결 없음 |
| DESIGN | 샘플 3점 (WALL·E) | 불교 실제 레퍼런스 0점 |
| Entity | 5개 | 주요 인물·장소·개념 대부분 미등록 |
| ATLAS changelog | 0.0.2~0.0.4만 기록 | 0.0.5~0.0.7 누락 |

## 2. 향후 개발 — 단기 (1~2개월)

지금 단계에서 가장 중요한 것은 **"완결된 하나"**를 만드는 것이다. 기능 확장보다 콘텐츠 하나를 끝까지 완성하는 경험이 파이프라인을 검증한다.

### 2.1 "붓다스토리 1장" 파이프라인 완주

```
목표: 붓다스토리 1장을 WORKS 문서 라이브러리와 BOOK/DESIGN으로 연결

  WORKS: 1장 문서 공개 (status: published)
    ↓
  DESIGN: 아난다·붓다·마야 턴어라운드 시트 1점씩
    ↓
  DESIGN: 룸비니·카필라바스투 배경 레퍼런스 1점씩
    ↓
  BOOK: 삽화가 포함된 완성 HTML 1권
    ↓
  BLOG: 제작 과정 기록 1편
```

이 파이프라인을 한 번 돌리면, 전체 시스템의 실제 병목과 워크플로우 문제가 드러난다. 기능 개발은 이 경험에서 나온 필요에 따라 진행한다.

### 2.2 기술적 필수 보완

**즉시 (이번 주)**

1. **Open Graph / 소셜 미리보기**
   - BaseLayout.astro에 og:title, og:description, og:image, twitter:card 추가
   - 각 페이지별 대표 이미지 지정
   - 카카오톡·텔레그램 등 한국 메신저 공유 시 미리보기 표시를 위해 필수

2. **sitemap.xml + robots.txt**
   - `@astrojs/sitemap` 통합 추가
   - Google Search Console 등록
   - 검색 엔진을 통한 자연 유입의 기초

3. **RSS 피드**
   - `@astrojs/rss`로 블로그 피드 생성
   - 불교 커뮤니티, 개발자 커뮤니티에서 구독 가능

4. **커스텀 404 페이지**
   - OL 브랜드에 맞는 404 페이지 (간결하고 고요한 디자인)
   - 홈·검색으로의 안내 링크

**1개월 내**

5. **이미지 최적화 파이프라인**
   - Astro `<Image />` 컴포넌트 도입
   - WebP 자동 변환, 반응형 srcset
   - DESIGN 페이지에서 특히 중요 (이미지가 핵심 콘텐츠)

6. **OLStats 컴포넌트 복원**
   - 현재 주석 처리된 통계 섹션을 동적 데이터로 복원
   - OLProductCards처럼 getCollection()으로 실시간 수치 반영

7. **ATLAS changelog 완성**
   - atlas.astro 사이드바의 CHANGELOG를 0.0.7까지 갱신
   - 향후 자동화: ATLAS 파일에서 빌드 메타데이터 추출

## 3. 향후 개발 — 중기 (3~6개월)

### 3.1 콘텐츠 확충 우선순위

불교 콘텐츠 기획자 관점에서, OL의 강점을 극대화할 콘텐츠 우선순위는 다음과 같다.

**1순위: 이야기 콘텐츠 (WORKS + BOOK)**

```
  붓다 스토리 시리즈 (10~12장 완결 목표)
    - 탄생 → 출가 → 고행 → 성도 → 초전법륜 → 교화 → 입멸
    - 각 장마다 삽화 2~3점, 인포그래픽 1점
    - 이것이 OL의 "대표작"이 된다
```

이야기 콘텐츠가 1순위인 이유:
- 선언문이 명시한 "몰입, 사유, 관조"는 정보 나열이 아닌 서사에서 발생한다
- 웹툰·애니메이션 확장(OL TOON)의 원작 역할
- 일반 대중 진입 장벽이 가장 낮은 형식
- DESIGN 레퍼런스 제작의 자연스러운 동기 부여 (이야기에 필요한 인물을 그린다)

**2순위: 디자인 레퍼런스 (DESIGN)**

```
  붓다스토리 캐릭터 시트 시리즈
    - 붓다, 아난다, 사리불, 목건련, 마하가섭 등 주요 제자 10인
    - 각 인물: 턴어라운드 + 표정 + 의상소품 (최소 3시트)
    
  배경 레퍼런스 시리즈
    - 기원정사, 죽림정사, 녹야원, 보드가야 등 주요 장소 5~8곳
    - 각 장소: 외관 + 평면도 (최소 2시트)
    
  소품 레퍼런스
    - 비구 육물, 법구(法具), 초기불교 시대 일상 소품
```

**3순위: 지식 콘텐츠 (Entity + Ontology)**

```
  주요 인물 Entity 30개
    - 붓다의 10대 제자, 주요 재가 신자, 외도 사상가
    
  주요 개념 Entity 20개
    - 사성제, 팔정도, 연기, 공, 무아 등
    
  주요 장소 Entity 15개
    - 팔대성지 + 주요 정사·마을
```

### 3.2 기능 개발 우선순위

**Entity 관계 시각화**

현재 Entity 시스템의 관계(relations)가 스키마에만 있고 시각화가 없다. 이것은 OL의 핵심 차별점이 될 수 있다.

```
구현 방향:
  - D3.js 또는 Cytoscape.js 기반 관계 그래프
  - 인물 간 관계 (스승-제자, 동문, 대립)
  - 인물-장소 관계 (어디서 활동했는가)
  - 인물-개념 관계 (어떤 가르침과 연결되는가)
  - OLGraphView.astro를 실제 구현으로 교체

주의:
  - 선언문의 "과시적 그래프, 복잡한 데이터 시각화는 OL의 길이 아니다"에 주의
  - 간결하고 고요한 시각화여야 한다
  - 사용자가 요청할 때만 표시 (기본은 텍스트 관계 목록)
```

**DESIGN 시리즈 묶음 뷰**

```
  같은 series 값을 가진 레퍼런스를 묶어서 보여주는 뷰
  예: "붓다스토리 캐릭터" 시리즈를 클릭하면
      해당 시리즈의 모든 시트가 한 페이지에 정렬
```

**WORKS ↔ BOOK 연결 UI**

```
  WORKS 문서가 BOOK 출판물과 연결될 때
  WORKS 페이지에서 "완결된 BOOK 보기 →" 링크 자동 표시
  BOOK 페이지에서 "관련 WORKS 문서 보기 →" 역링크
```

**다크 모드 이미지 대응**

```
  DESIGN 이미지 중 밝은 배경 이미지는 다크 모드에서 눈부심
  이미지 컨테이너에 배경색 고정 또는 미디어 쿼리 대응 필요
```

## 4. 향후 개발 — 장기 (6개월~1년)

### 4.1 OL TOON 페이지

기획서에 명시된 OL TOON은 별도 웹페이지가 필요하다고 했지만, 초기에는 OL HOME 안에 `/toon` 경로로 시작하는 것이 효율적이다.

```
구현 방안:
  - src/pages/toon/index.astro — 웹툰 목록
  - src/pages/toon/[slug].astro — 개별 에피소드 뷰어
  - 세로 스크롤 웹툰 뷰어 (한국 웹툰 표준 UX)
  - 이미지는 public/toon/{시리즈명}/{에피소드}/ 에 저장
  - 콘텐츠가 충분해지면 별도 도메인으로 분리
  
콘텐츠 전략:
  - 붓다스토리 1장을 웹툰 형식으로 변환하는 것이 첫 실험
  - DESIGN 레퍼런스가 있어야 웹툰 작화의 일관성 확보 가능
  - 따라서 DESIGN → TOON 순서가 맞다
```

### 4.2 국제화(i18n) — 영문 페이지

불교는 본질적으로 국제적 종교이며, OL의 CC0 정신은 언어 장벽이 없을 때 가장 빛난다.

```
1단계: 영문 랜딩 페이지 (index 페이지만)
2단계: 영문 선언문, 브랜드 소개
3단계: Entity 영문 병행 (이미 name.en 필드가 스키마에 있음)
4단계: BOOK 영문 번역 (장기)

기술적 접근:
  - Astro의 i18n 라우팅 활용 (/en/... 경로)
  - 또는 별도 영문 랜딩 페이지만 최소 구현
```

### 4.3 PWA / 오프라인 지원

OL의 "오프라인 우선" 철학과 직접 맞닿는 기능이다.

```
  - Service Worker로 한 번 방문한 페이지 오프라인 캐싱
  - BOOK HTML 파일의 다운로드 없이도 오프라인 열람 가능
  - manifest.json으로 홈 화면 설치 지원
  - 선언문 10조 "가장 가볍게, 가장 널리"의 기술적 구현
```

### 4.4 검색 강화

```
  현재 Pagefind 기반 전문 검색이 구현되어 있지만:
  - Entity 기반 연관 검색 (검색 결과에 관련 인물·개념 표시)
  - 빨리어·산스크리트어·한자 병행 검색
  - 태그 기반 크로스 컬렉션 검색 (같은 태그를 가진 WORKS+DESIGN+BLOG 통합 결과)
```

## 5. 운영 조언

### 5.1 콘텐츠 제작 리듬

```
주간 리듬 (제안):
  월~수: WORKS 문서 작업 (1편 또는 1장 분량)
  목~금: DESIGN 레퍼런스 1점 제작
  토:    BLOG 작업 기록 1편
  일:    빌드 → 배포 → 확인

월간 목표:
  WORKS: 4편
  DESIGN: 4~8점
  BLOG: 4편
  BOOK: 분기 1권 완결 목표
```

이 리듬이 중요한 이유: OL은 1인 프로젝트이므로 "지속 가능한 제작 속도"가 핵심이다. 선언문이 경계한 "생산성 SaaS"의 함정에 제작자 자신이 빠지지 않아야 한다.

### 5.2 품질 기준

```
WORKS 발행 기준:
  - 원전 출처가 명시되어 있는가
  - 현대어로 자연스러운가
  - Entity와 연결되어 있는가 (최소 1개)

DESIGN 발행 기준:
  - 해상도 3000px 이상인가
  - type, series, scriptureRef가 채워져 있는가
  - 다른 시트와 등신·스타일이 일관적인가

BOOK 발행 기준:
  - 관련 WORKS 문서와 출처 관계가 정리되어 있는가
  - 삽화가 최소 1점 포함되어 있는가
  - ATLAS에서 정상 동작하는가
```

### 5.3 커뮤니티 전략

OL은 "평가 기능, 댓글, 소셜"을 절대 추가하지 않겠다고 선언했다. 이것은 존중되어야 한다. 하지만 "사용자와의 접점"은 다른 방식으로 만들 수 있다.

```
제안:
  1. GitHub Issues를 피드백 채널로 명시
     - 홈페이지 푸터에 "피드백·기여" 링크로 GitHub Issues 연결
     - 이미 구현되어 있으나 더 적극적으로 안내
  
  2. 뉴스레터 (선택적)
     - 월간 또는 분기 단위
     - 이메일 수집 → 발송만 (구독 통계·분석 없음)
     - 선언문과 충돌하지 않는 최소한의 "알림"
  
  3. 불교 커뮤니티 연계
     - partners 페이지에 등록된 사찰·단체에 실물 공유
     - BOOK HTML 파일을 USB·이메일로 전달하는 오프라인 배포
     - 법회·수련회에서 ATLAS 시연
  
  4. 오픈소스 기여 안내
     - CONTRIBUTING.md 작성
     - "번역에 참여하기", "삽화 기여하기" 등 구체적 기여 경로 안내
     - CC0 라이선스의 의미를 일반인도 이해할 수 있게 설명
```

### 5.4 "하지 말아야 할 것" 재확인

개발이 진행되면서 기능 추가의 유혹이 생길 때, 선언문 부록 A를 다시 읽어야 한다.

```
특히 주의할 유혹들:

  "방문자 수를 알고 싶다" 
    → Google Analytics 등 텔레메트리 추가 금지 (선언문 위반)
    → 대안: GitHub Pages 기본 트래픽 통계 (저장소 settings에서 확인)

  "댓글 기능이 있으면 좋겠다"
    → 평가 기능 추가 금지 (선언문 위반)
    → 대안: GitHub Discussions 또는 이메일 피드백

  "AI 챗봇을 넣으면 어떨까"
    → AI의 능동적 개입 금지 (선언문 위반)
    → 대안: ATLAS 내부에서만, 요청 기반으로만

  "진도율을 보여주면 동기부여가 될 텐데"
    → 게이미피케이션 금지 (선언문 위반)
    → 대안: 없음. 이것은 의지적 결정이다.
```

## 7. OL의 본래 취지를 가장 잘 발휘하는 방향

### 7.1 핵심 질문으로 돌아가기

선언문의 권두언:

> *"사람들이 이 경험을 통해 정말 더 자유로워지는가?"*

이 질문에 비추어 볼 때, 지금 OL에게 가장 필요한 것은 기능이 아니라 **콘텐츠**이다.

아무리 아름다운 도서관이라도 책이 비어 있으면 아무도 오지 않는다. OL HOME은 이미 "도서관"으로서 충분한 구조를 갖추었다. 이제 필요한 것은 **책장을 채우는 것**이다.

### 7.2 "첫 번째 완결된 독서 경험" 만들기

```
가장 임팩트 있는 다음 단계:

  누군가가 OL에 와서
  붓다스토리 1장을 읽고
  삽화를 보고
  관련 인물의 Entity 페이지를 둘러보고
  ATLAS에서 파일을 내려받아 오프라인에서 다시 읽고
  "이것을 누군가에게 보내고 싶다"고 느끼는 것.

  이 경험이 한 번이라도 완성되면
  OL은 "작동하는 프로젝트"가 된다.
```

### 7.3 창의적 확장 가능성

OL이 가진 독특한 잠재력은 "도구와 콘텐츠를 함께 배포한다"는 점이다. 이 원리를 확장하면:

**"불교 콘텐츠 제작 키트" 배포**

```
  DESIGN 레퍼런스 + ATLAS 빈 틀 + 제작 매뉴얼을
  하나의 패키지로 묶어 배포

  받는 사람은:
  - 레퍼런스를 보고 자기만의 삽화를 그리고
  - ATLAS에 자기만의 콘텐츠를 담고
  - 다시 누군가에게 보낸다

  이것이 선언문의 "한 사람으로 끝나지 않는 콘텐츠"의 실현이다
```

**사찰·수련회 연계 콘텐츠**

```
  partners 페이지에 등록된 사찰과 협력하여:
  - 해당 사찰의 역사를 BOOK으로 제작
  - 사찰 건축을 DESIGN 레퍼런스로 기록
  - 법회에서 ATLAS를 활용한 설법 자료 제공

  OL의 콘텐츠가 화면 안에만 머물지 않고
  실제 수행 공간과 연결되는 것
```

**"고요한 웹" 운동**

```
  OL의 선언문이 제시하는 가치 —
  추적 없음, 광고 없음, 게이미피케이션 없음, 로컬 우선 —
  이것은 불교 프로젝트를 넘어
  "인터넷은 이래야 한다"는 하나의 선언이 될 수 있다.

  /pages/manifesto를 영문으로 번역하여
  국제 오픈소스·디지털 윤리 커뮤니티에 공유하는 것도 의미 있다.
```

## 9. 마무리

OL은 기술 프로젝트이면서 동시에 수행 프로젝트이다. 코드를 쓰는 것이 곧 경전을 필사하는 마음이고, 레퍼런스를 그리는 것이 곧 불화(佛畫)를 조성하는 마음이다.

이 문서의 모든 기술적 제안은 하나의 전제 위에 있다:

> *콘텐츠를 통해 한 사람이라도 더 자유로워지는가.*

이 질문에 "예"라고 답할 수 있는 방향이면 옳고, 아니라면 아무리 기술적으로 훌륭해도 OL의 길이 아니다.
