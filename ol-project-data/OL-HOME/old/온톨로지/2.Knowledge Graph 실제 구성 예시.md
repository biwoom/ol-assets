# Knowledge Graph 실제 구성 예시

예를들어 
파일: "부처님의 탄생.md"
내용: 마야부인이 룸비니 동산 무우수 나무가지를 붙잡고 태자를 낳으셨다.
태그: #마야부인, #룸비니, #무우수, #태자, #탄생, #싯다르타.
태그를 다는 일과 Knowledge Graph 는 어떻게 다른가?
json등으로 각 md 파일을 연결하나?

---

아주 중요한 질문입니다.
사실 대부분 사람들이 여기서 막힙니다.

태그(`#탄생`, `#붓다`)는 이해하기 쉽지만,
Knowledge Graph는:

> “태그를 넘어서 관계 자체를 데이터화”

하는 것입니다.

그리고 당신이 구상하는 OL 같은 시스템은 사실상:

```text id="2i7xqs"
Markdown 문서 + 메타데이터 + 관계 정의
```

의 조합으로 매우 자연스럽게 구축 가능합니다.

# 1. 가장 단순한 단계: 태그 시스템

예를 들어:

## 파일

```md id="7l6ogz"
부처님의 탄생.md
```

## 내용

```md id="dwb4uq"
마야부인이 룸비니 동산 무우수 나무가지를 붙잡고
태자를 낳으셨다.
```

## 태그

```md id="u2cku0"
#마야부인
#룸비니
#무우수
#태자
#탄생
#싯다르타
```

이것은:

```text id="3xkxpj"
문서 ↔ 키워드
```

수준입니다.

즉:

```text id="9r3r39"
[부처님의 탄생.md]
   ├─ #마야부인
   ├─ #룸비니
   ├─ #탄생
```

정도.

이것은 아직 “의미 관계”가 없습니다.

---

# 2. Knowledge Graph 단계로 가면?

여기서는 단순 태그가 아니라:

```text id="znd27d"
누가?
어디서?
무엇을?
어떤 관계로?
```

를 구조화합니다.

즉:

| 요소   | 의미          |
| ---- | ----------- |
| 마야부인 | 인물(Person)  |
| 룸비니  | 장소(Place)   |
| 태자   | 인물(Person)  |
| 탄생   | 사건(Event)   |
| 무우수  | 식물/상징(Tree) |

그리고 관계:

```text id="rf7jjl"
마야부인 ─ 출산했다 → 태자
태자 ─ 출생했다 → 룸비니
탄생 사건 ─ 발생장소 → 룸비니
탄생 사건 ─ 관련인물 → 마야부인
```

---

# 3. 핵심: "문서 자체"와 "개체(Entity)"를 분리

Knowledge Graph에서 매우 중요한 점:

> 문서와 개체는 다르다.

---

## 문서(Document)

```text id="0p7zv8"
부처님의 탄생.md
```

은 하나의 “설명 문서”입니다.

---

## 개체(Entity)

반면:

* 마야부인
* 룸비니
* 싯다르타
* 탄생 사건

은 세계를 구성하는 “실체(Entity)”입니다.

Knowledge Graph는:

```text id="fc8ivg"
문서 중심
→ 개체 중심
```

으로 사고가 이동합니다.

---

# 4. 실제 구현 방식 (가장 중요)

여기서 여러 방법이 있습니다.

---

# 방법 A — Markdown Frontmatter 방식 (추천)

이게 현재 가장 현실적이고 OL과도 잘 맞습니다.

---

## 예시

```md id="yxd6qy"
---
id: event-buddha-birth
type: event

title: 부처님의 탄생

persons:
  - maya
  - siddhartha

places:
  - lumbini

concepts:
  - birth

symbols:
  - sala-tree

relations:
  - subject: maya
    predicate: gave_birth_to
    object: siddhartha

  - subject: siddhartha
    predicate: born_in
    object: lumbini

---

마야부인이 룸비니 동산 무우수 나무가지를 붙잡고
태자를 낳으셨다.
```

---

# 5. 이 구조의 핵심

여기서:

```yaml id="l2td5d"
subject: maya
predicate: gave_birth_to
object: siddhartha
```

가 바로 Knowledge Graph의 triple입니다.

즉:

```text id="24y1li"
마야부인 → 출산했다 → 싯다르타
```

---

# 6. Entity 사전(Entity Registry)

그리고 각 entity는 별도 정의 파일을 둡니다.

예:

---

## maya.md

```md id="ekz3ny"
---
id: maya
type: person
name: 마야부인

aliases:
  - 摩耶夫人
  - Māyā

related_places:
  - kapilavastu

---

석가모니의 어머니.
```

---

## lumbini.md

```md id="65ozfh"
---
id: lumbini
type: place

modern_country: Nepal
latitude: ...
longitude: ...
---

붓다 탄생지.
```

---

# 7. 이제 그래프가 생긴다

시스템은 모든 md 파일의 frontmatter를 읽고:

```text id="xta6go"
maya
siddhartha
lumbini
```

등의 관계를 자동 연결합니다.

즉:

```text id="qg1lkg"
[마야부인]
   └─ gave_birth_to → [싯다르타]

[싯다르타]
   └─ born_in → [룸비니]
```

---

# 8. JSON 파일로 따로 관리할 수도 있나?

가능합니다.

---

## 예시

```json id="w1i5j5"
{
  "entities": [
    {
      "id": "maya",
      "type": "person"
    }
  ],

  "relations": [
    {
      "subject": "maya",
      "predicate": "gave_birth_to",
      "object": "siddhartha"
    }
  ]
}
```

---

하지만 OL 프로젝트에서는 나는 JSON-only 방식은 비추천입니다.

왜냐하면:

```text id="z5db3u"
JSON 중심
→ 인간이 읽기 어려움
→ 저작 경험이 경직됨
```

반면 Markdown + frontmatter는:

```text id="z6lqyo"
사람도 읽기 쉬움
AI도 처리 쉬움
Git 친화적
장기보존성 높음
```

입니다.

---

# 9. OL에 특히 적합한 구조

당신 프로젝트는 이미:

```text id="drx74e"
문헌 중심
```

이므로,

가장 좋은 방식은:

```text id="y9db9g"
Markdown 문서
+ YAML frontmatter
+ Entity Registry
+ Relation triples
```

조합입니다.

이게 현재 디지털 인문학에서도 매우 강력한 패턴입니다.

---

# 10. 실제 OL 구조 예시

예를 들어:

```text id="h8z24m"
/content
  /events
    buddha-birth.md

/entities
  /persons
    maya.md
    siddhartha.md

  /places
    lumbini.md

  /concepts
    birth.md
```

---

# 11. 그러면 실제로 무엇이 가능해지나?

이제 시스템은 자동으로:

---

## 자동 연결

```text id="xwquxg"
마야부인 등장 문서 모두 보기
룸비니 관련 사건 보기
싯다르타 생애 타임라인
```

---

## 자동 백링크

```text id="quprcc"
이 장소가 언급된 문서
이 인물이 등장하는 장면
```

---

## 자동 지도 생성

```text id="c1m7wf"
룸비니
→ 보드가야
→ 녹야원
→ 쿠시나가라
```

불교 성지 순례 경로.

---

## 자동 계보도

```text id="9c9lm8"
붓다
 ├─ 사리불
 ├─ 목건련
 └─ 아난다
```

---

# 12. 중요한 점: 태그와 관계는 다르다

이 차이가 핵심입니다.

---

## 태그

```text id="stpw0u"
#룸비니
```

= 그냥 키워드.

---

## 관계

```text id="j9n7r3"
싯다르타 ─ born_in → 룸비니
```

= 의미가 명확함.

Knowledge Graph는 후자를 사용합니다.

---

# 13. OL은 처음부터 거대한 Graph DB가 필요하지 않다

이것도 중요합니다.

많은 사람들이:

```text id="x6ltgp"
Neo4j
RDF
SPARQL
```

같은 거대한 시스템부터 생각합니다.

하지만 OL 초기 단계에서는 오히려:

```text id="l3q1t7"
Markdown + YAML + JSON export
```

만으로도 충분합니다.

---

# 14. 현실적 추천 (OL 프로젝트 기준)

당신에게 가장 추천하는 초기 구조:

| 단계        | 추천               |
| --------- | ---------------- |
| 문서        | Markdown         |
| 메타데이터     | YAML frontmatter |
| Entity ID | 고유 문자열           |
| 관계        | triple 구조        |
| 저장        | 파일 기반            |
| 검색        | JS index         |
| 시각화       | 최소화              |
| UX        | 조용한 문맥 연결        |

---

# 15. OL 철학과 가장 잘 맞는 방식

특히 중요한 점:

OL 선언문은:

> “그래프를 과시하지 않는다”
> “깊은 체류와 문맥”

를 중시합니다. 

따라서 OL의 Knowledge Graph는:

```text id="fru5n6"
백엔드에서는 치밀한 연결망
프론트에서는 자연스러운 독서 흐름
```

형태가 가장 적합합니다.

즉 사용자는:

```text id="xy2q4m"
'그래프를 본다'
```

보다,

```text id="d8w5w0"
'맥락이 자연스럽게 이어진다'
```

를 경험하게 됩니다.

이건 실제로 매우 수준 높은 방향입니다.
