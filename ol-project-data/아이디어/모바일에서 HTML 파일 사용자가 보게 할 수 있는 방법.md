모바일 환경에서 서버(GitHub Pages 등)를 거치지 않고, 사용자가 다운로드한 단일 HTML 파일(Single File Web App)을 직접 열어보게 하는 것은 모바일 OS(iOS/Android)의 강력한 보안 샌드박스(Sandbox) 정책 때문에 PC만큼 간단하지 않습니다.

이 문제를 해결하기 위해 현재 활용 가능한 접근 방식들을 **객체 지향적이고 실용적인 관점**에서 분석하여 세 가지 레이어로 분류했습니다.

## 1. 네이티브 앱 전환 (가장 안정적인 배포)

HTML 파일을 패키징하여 앱스토어나 구글 플레이스토어에 등록하거나, APK/IPA 파일로 직접 배포하는 방식입니다. 오프라인 구동이 완벽히 보장되며 사용자 경험이 가장 뛰어납니다.

### Capacitor / Cordova (하이브리드 앱 패키징)

기존 HTML/CSS/JS 소스를 그대로 유지한 채 웹뷰(WebView)를 감싸는 네이티브 래퍼(Wrapper)를 씌우는 방식입니다.

- **장점:** 단일 HTML 파일과 자산(Assets)이 앱 내부 로컬 경로(`file://` 또는 `localhost`)로 내장되므로 인터넷 연결이 전혀 필요 없습니다.
    
- **단점:** 빌드를 위해 개발 환경(Xcode, Android Studio) 세팅이 필요하며, 마켓 배포 시 심사 과정을 거쳐야 합니다.
    

### PWA (Progressive Web App) 방식의 우회

엄밀히 말하면 '단일 파일 열기'는 아니지만, 최초 1회만 웹으로 접속하면 서비스 워커(Service Worker)가 HTML 전체를 로컬 스토리지에 캐싱합니다.

- **작동 방식:** 사용자가 모바일 브라우저에서 '홈 화면에 추가'를 누르면 오프라인에서도 아이콘을 통해 앱처럼 구동됩니다.
    
- **적합성:** 실질적으로 오프라인 단일 앱처럼 동작하므로 파일 배포의 번거로움을 완전히 지울 수 있는 가장 현대적인 대안입니다.
    

## 2. 모바일 로컬 웹 서버 앱 활용 (사용자 직접 실행)

사용자가 모바일 기기에 파일 접근 권한을 가진 특정 앱을 설치하고, 그 앱을 통해 HTML을 구동하는 방식입니다. 개발자가 별도 앱을 빌드할 필요가 없을 때 유용합니다.

### Android 환경

안드로이드는 파일 시스템 접근이 비교적 자유로워 상대적으로 구현이 쉽습니다.

- **로컬 웹 서버 앱 사용 (추천):** `Http Canterbury` 또는 `Simple HTTP Server` 같은 앱을 마켓에서 설치한 뒤, HTML 파일이 있는 폴더를 루트로 지정하면 로컬 호스트(`http://localhost:8080`)를 통해 모바일 크롬에서 완벽하게 구동됩니다.
    
- **HTML 뷰어 앱:** `HTML Viewer` 계열의 앱으로 파일을 열 수 있으나, 보안 정책(CORS) 때문에 외부 JS 모듈이나 로컬 이미지 로드가 차단될 수 있으므로 모든 스크립트와 스타일이 HTML 한 파일 내에 내장(Inlined)되어 있어야 합니다.
    

### iOS (iPhone/iPad) 환경

iOS는 보안상 브라우저가 로컬 파일 시스템(`file://`)의 스크립트 실행을 엄격히 차단합니다. 따라서 일반적인 '파일' 앱에서 HTML을 누르면 텍스트나 정적 페이지로만 보이고 JS가 먹통이 됩니다.

- **Documents by Readdle 앱 활용:** iOS에서 로컬 웹앱을 구동하는 가장 표준적인 우회책입니다. 이 앱 내부에 HTML 파일을 넣고 내장 브라우저로 실행하면, 스크립트가 정상 작동하는 단일 오프라인 뷰어로 활용할 수 있습니다.

> 테스트 결과 잘 작동함. 

## 3. 단일 파일 웹북 제작 시 기술적 필수 고려사항

HTML 파일을 직접 배포하는 방식을 채택할 경우, 브라우저의 보안 정책(CORS 및 로컬 스토리지 제한)을 회피하기 위해 소스코드 레벨에서 다음 제약 조건을 반드시 준수해야 합니다.

- **모든 자산의 인라인화(Inlining):** CSS, JavaScript, 이미지(Base64 인코딩)를 별도 파일로 분리하지 말고 하나의 `.html` 파일 안에 전부 포함해야 합니다. 로컬 파일 시스템에서는 상대 경로(`./script.js`) 참조가 차단될 확률이 매우 높습니다.
    
- **ES 모듈 사용 지양:** `<script type="module">` 은 로컬 환경(`file://`)에서 CORS 에러를 발생시킵니다. 웹팩(Webpack)이나 바이테(Vite) 같은 번들러를 사용한다면 single-file 플러그인을 통해 구형 번들 방식(IIFE 형태)으로 빌드해야 합니다.
    
- **데이터 저장소 전략:** `localStorage`나 `IndexedDB`는 `file://` 경로에서 브라우저 및 OS별로 동작 여부가 판이합니다. 독서 진도나 설정을 저장해야 한다면, 상태를 텍스트 파일(JSON)로 내보내고 가져오는(Import/Export) 기능을 앱 내에 구현하는 것이 가장 안전합니다.
    

### 💡 전문가적 제언

배포 대상이 **불특정 다수**라면 **PWA** 형태로 개발하여 최초 1회 홈 화면 추가를 유도하는 것이 유지보수와 사용자 경험 측면에서 압도적으로 유리합니다.

반면 기업 내부용이거나 **완전한 폐쇄망 오프라인 환경**을 원하신다면, 단일 HTML 파일을 **Capacitor**를 통해 하이브리드 앱(APK/IPA)으로 빌드하여 배포하는 가공 공정을 거치기를 권장합니다.
