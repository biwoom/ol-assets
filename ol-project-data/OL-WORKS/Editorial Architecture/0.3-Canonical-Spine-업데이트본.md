## 0. 이번 업데이트의 핵심

이전 문서에서 Canonical Spine은 주로 『불교스토리』의 **기준 목차·사건 배열·원전 매핑표**로 설명되었다. 이 정의는 여전히 유효하다. 다만 OL HOME의 최신 구조가 정리되면서 Canonical Spine의 역할은 더 넓어졌다.

이제 Canonical Spine은 단순한 편집용 목차가 아니라 다음 세 층을 동시에 담당해야 한다.

```text
1. Editorial Spine   = 원전 비교·번역·합본을 위한 편집 뼈대
2. Production Spine  = WORKS → BOOK → DESIGN → BLOG 제작 파이프라인의 기준
3. Semantic Spine    = Entity / Ontology / Knowledge Graph의 event registry 초안
```

따라서 Canonical Spine은 『불교스토리』의 원고 제작 과정뿐 아니라 OL HOME의 `src/content/works`, `src/content/book`, `src/content/design`, `src/content/entities`, `src/content/ontology`, 그리고 `public/books`, `public/design`까지 이어지는 **콘텐츠 운영 좌표계**가 되어야 한다.

# 1. Canonical Spine의 업데이트된 정의

**Canonical Spine**은 『불교스토리』 전체 프로젝트의 “기준 목차이자 사건 배열의 중추”이다. 그러나 OL 최신 구조에서는 이것을 다음처럼 더 정확히 정의한다.

> Canonical Spine은 4종 원전 소스를 공통 사건 단위로 정렬하고, 그 사건을 번역·합본·삽화·BOOK 출판·Entity 등록·Knowledge Graph 확장까지 연결하기 위한 OL식 기준 인덱스이다.

비유하면 다음과 같다.

```text
Canonical Spine       = 부처님 일대기 프로젝트의 척추
Event ID              = 척추 마디이자 Knowledge Graph의 event 후보 ID
Source Segment        = 각 마디에 붙는 원전 조각
Translation Layer     = 각 조각의 한국어 번역층
Narrative Merge       = 척추를 따라 구성된 최종 한국어 서사
WORKS                 = 제작 중인 원고 공개 작업장
BOOK                  = 완결된 단일 HTML 출판물
DESIGN                = 사건·인물·장소·소품의 시각 레퍼런스 아카이브
Entity/Ontology       = 사건·인물·장소·개념을 연결하는 의미망
OL HOME               = 전체 콘텐츠와 의미망을 모으는 Index Layer
```

이전 문서의 핵심 구조인 `Season → Episode → Sequence → Event → Segment`는 그대로 유지한다. 다만 이제 각 Event는 단순한 편집 단위가 아니라 다음 네 가지로 동시에 작동한다.

```text
Event = 원전 비교 단위
Event = WORKS 원고 단위
Event = 삽화·웹툰·영상화 단위
Event = Knowledge Graph의 event entity 후보
```

# 2. OL 최신 구조에서 Canonical Spine이 연결해야 할 대상

OL HOME은 Astro 기반 정적 사이트이며 콘텐츠는 `src/content/`의 Markdown 프론트매터와 `public/`의 실제 파일로 분리된다. 따라서 Canonical Spine은 다음 저장 구조와 맞물려야 한다.

```text
ol-home/
├── src/content/
│   ├── works/       ← 제작 중 원고: 번역 초안, 합본 초안, 주석 연구
│   ├── book/        ← 완성 BOOK의 메타데이터 md
│   ├── design/      ← 이미지·레퍼런스 메타데이터 md
│   ├── entities/    ← 인물·장소·개념·사건 entity registry
│   └── ontology/    ← 관계 유형과 온톨로지 정의
│
└── public/
    ├── books/       ← 완성 BOOK HTML
    └── design/      ← 실제 이미지 파일
```

따라서 『불교스토리』 Canonical Spine은 다음과 같이 실제 사이트 구조에 대응해야 한다.

```text
Canonical Spine
  ↓
WORKS 원고
  src/content/works/buddha-story/{episode-or-sequence}.md
  ↓
BOOK 완성본
  public/books/buddha-story/index.html
  src/content/book/buddha-story.md
  ↓
DESIGN 레퍼런스
  public/design/{event-or-entity-image}.png
  src/content/design/{event-or-entity-image}.md
  ↓
Entity / Ontology
  src/content/entities/{entity-id}.md
  src/content/ontology/{ontology-file}.md
  ↓
Global Index Layer
  OL HOME 검색·목록·관계·향후 그래프
```

이 점 때문에 Canonical Spine의 ID 설계는 단순한 내부 작업 편의를 넘어서 **홈페이지 URL, 파일명, 엔티티 ID, 이미지 메타데이터, BOOK 메타데이터**와 충돌하지 않아야 한다.

# 3. Canonical Spine의 목표 — 업데이트판

Canonical Spine의 목표는 기존 네 가지에 두 가지를 추가하여 여섯 가지로 정리한다.

## 3.1 기준 순서 확정

The Great Chronicle of Buddhas의 목차를 기본 뼈대로 삼되, 그것을 기계적으로 복사하지 않고 『불본행집경』·『불본행경』의 사건과 설화를 붙일 수 있는 구조로 재설계한다.

중요한 점은 GCB가 “base spine”이지 “final authority”는 아니라는 것이다.

```text
The Great Chronicle of Buddhas = 목차와 서사 진행의 기본 기준
불본행집경·불본행경       = 한역 불전 전승의 보강·변이·확장 자료
최종 원고                 = 네 원전을 바탕으로 한 현대 한국어 편집본
```

## 3.2 사건 단위화

부처님 일대기는 단순 장절 목차가 아니라 사건들의 연쇄이다. 예를 들어 “탄생”이라는 큰 장 안에도 다음 사건들이 분리된다.

```text
도솔천 관찰
마야부인의 태몽
룸비니로 향함
무우수 아래에 섬
태자의 탄생
칠보행
탄생 선언
아시타 선인의 예언
```

이 사건 단위가 이후 원전 비교, 번역, 삽화, Entity 등록의 기준이 된다.

## 3.3 원전 매핑 기준 마련

4종 원전은 같은 사건을 같은 방식으로 다루지 않는다. 어떤 사건은 영어 원고에 자세하고, 어떤 사건은 한역에만 풍부하며, 어떤 사건은 한쪽에는 없고 다른 쪽에는 있다. Canonical Spine은 이 차이를 수용하는 공통 좌표계다.

## 3.4 후속 제작 자산의 기준점 확보

Event ID가 확정되면 다음 자산들이 같은 ID에 연결된다.

```text
원전 segment
직역층
의미 번역층
최종 합본 원고
용어집
variant note
reference note
삽화
캐릭터·장소·소품 레퍼런스
OL BOOK 카드 또는 문서
OL TOON 컷
OL STUDIO 영상 대본
Entity / relation triples
```

## 3.5 OL HOME 등록 구조와 연결

Canonical Spine은 WORKS와 BOOK 등록을 염두에 두어야 한다. 제작 중 원고는 `src/content/works/buddha-story/`에 등록되고, 완성본은 `public/books/buddha-story/index.html` 및 `src/content/book/buddha-story.md`에 등록된다.

따라서 Canonical Spine은 최소한 다음 메타데이터를 지원해야 한다.

```yaml
works:
  series: "붓다 스토리"
  category: "인물 · 행적"
  status: draft
  order: 1
  published: false

book:
  htmlPath: "buddha-story"
  version: "v1.0"
  status: ready
  published: true
```

## 3.6 Semantic Export의 기준 제공

OL BOOK은 독립 단일 HTML로 유지하되, OL HOME은 전체 BOOK의 metadata/index/graph를 수집하는 Index Layer가 된다. 따라서 Canonical Spine은 장기적으로 `.meta.json` 또는 동등한 semantic export를 생성할 수 있어야 한다.

```text
OL BOOK
  ├─ single HTML
  ├─ local metadata
  └─ exportable semantic metadata

OL HOME
  ├─ entity registry
  ├─ ontology
  ├─ cross-book relation
  ├─ global graph
  └─ semantic search
```

# 4. Canonical Spine은 “목차”가 아니라 세 겹의 편집 지도다

기존 문서에서는 Canonical Spine을 “목차가 아니라 편집 지도”라고 설명했다. 최신 OL 구조에서는 이 지도를 세 겹으로 나누어 이해하는 것이 좋다.

## 4.1 Editorial Map

원전 비교·번역·합본을 위한 지도다.

```yaml
event_id: BL-EV-012
canonical_title_ko: 마야부인의 태몽
canonical_title_en: Queen Māyā's Dream
episode_id: BL-EP-02
sequence_id: BL-SEQ-02-01
source_presence:
  gcb_original: present
  gcb_revised: present
  fobenxingji_jing: present
  fobenxing_jing: partial
variant_risk: high
needs_human_review: true
```

## 4.2 Production Map

WORKS, BOOK, DESIGN, BLOG 제작을 위한 지도다.

```yaml
production:
  works_path: "src/content/works/buddha-story/02-01-maya-dream.md"
  book_target: "public/books/buddha-story/index.html"
  design_needed:
    - "maya-dream-illustration"
    - "maya-character-reference"
  blog_note: "제작 과정 기록 후보"
```

## 4.3 Semantic Map

Entity와 Knowledge Graph를 위한 지도다.

```yaml
semantic:
  event_entity_id: "event-maya-dream"
  persons:
    - maya-devi
    - siddhartha-gautama
  places:
    - kapilavastu
  concepts:
    - conception
    - auspicious-dream
  symbols:
    - white-elephant
  relations:
    - subject: maya-devi
      predicate: dreamed_of
      object: white-elephant
    - subject: event-maya-dream
      predicate: precedes
      object: event-buddha-birth
```

이 세 지도를 분리해 두면, AI 에이전트가 구조화 작업을 하더라도 편집 판단, 제작 일정, 의미 관계가 뒤섞이지 않는다.

# 5. 단위 체계 — 최신 OL 구조에 맞춘 정리

기본 단위 체계는 다음과 같다.

```text
Season   = 불교스토리 전체 프로젝트 / 한 권의 큰 작품
Episode  = 큰 장 / WORKS 시리즈 내 장 / BOOK 목차 대단위
Sequence = 독립 단막 / WORKS 개별 원고 후보 / BOOK 문서 후보
Event    = 사건 단위 / 원전 비교 단위 / KG event 후보
Segment  = 각 원전에서 잘라낸 대응 원문 조각
Cut      = 삽화·웹툰·영상화를 위한 문장 또는 장면 호흡
```

OL BOOK의 단위 체계와 연결하면 다음과 같다.

| Canonical Spine | OL BOOK/WORKS | DESIGN/TOON | Ontology |
|---|---|---|---|
| Season | Book | 전체 작품 스타일 | book entity |
| Episode | Chapter | 회차·장면군 | narrative section |
| Sequence | Document / WORKS md | 단막·콘티 묶음 | document node |
| Event | Scene | 삽화·컷 후보 | event entity |
| Segment | Source block | 고증 자료 | source reference |
| Cut | Sentence | 컷·대사·내레이션 | optional annotation |

# 6. Event ID 설계 원칙 — 업데이트판

Event ID는 앞으로 여러 위치에서 재사용된다. 따라서 사람이 읽기 쉽고, 파일명으로도 안전하며, 장기적으로 바꾸지 않는 구조가 좋다.

## 6.1 권장 형식

```text
bl-ev-0001-sumedha-meets-dipankara
bl-ev-0024-buddha-birth-at-lumbini
bl-ev-0031-asita-prophecy
```

규칙:

```text
bl      = Buddha Life 또는 Buddha Story Life
 ev     = Event
 0001   = 정렬을 위한 고정 번호
 slug   = 의미를 나타내는 영문 소문자 slug
```

## 6.2 ID와 제목 분리

제목은 나중에 바뀔 수 있다. ID는 되도록 바꾸지 않는다.

```yaml
event_id: "bl-ev-0024-buddha-birth-at-lumbini"
canonical_title_ko: "룸비니에서 태어나다"
working_title_ko: "부처님의 탄생"
final_title_ko: "무우수 아래에서"
```

## 6.3 Entity ID와의 관계

Event ID는 곧바로 entity ID로도 쓸 수 있다.

```yaml
entity:
  id: "event-buddha-birth-at-lumbini"
  type: event
  canonical_event_id: "bl-ev-0024-buddha-birth-at-lumbini"
```

초기에는 Event ID와 Entity ID를 완전히 동일하게 해도 되지만, 장기적으로는 다음처럼 역할을 구분하는 것이 더 안전하다.

```text
canonical_event_id = 제작·편집용 ID
entity_id          = 지식그래프용 ID
slug               = URL·파일명용 ID
```

# 8. OL HOME 등록 구조와의 대응

## 8.1 WORKS 대응

WORKS는 완결 이전의 초고, 번역 초안, 주석 연구를 공개하는 작업장이다. 따라서 Canonical Spine의 Sequence 또는 Episode는 WORKS md 파일로 연결된다.

권장 구조:

```text
src/content/works/buddha-story/
  00-premise.md
  01-sumedha-vow.md
  02-birth.md
  03-youth.md
  04-renunciation.md
```

WORKS frontmatter 예시:

```yaml
---
title: "2장 — 탄생: 도솔천에서 룸비니까지"
series: "붓다 스토리"
```

# 9. Knowledge Graph 반영 원칙

## 9.1 태그와 관계를 구분한다

태그는 키워드다.

```text
#룸비니
#마야부인
#탄생
```

관계는 의미 구조다.

```text
마야부인 ─ gave_birth_to → 싯다르타
싯다르타 ─ born_in → 룸비니
탄생 사건 ─ occurred_at → 룸비니
```

Canonical Spine은 태그 목록만 만들면 안 된다. 최소한 주요 사건에는 관계 triples가 붙어야 한다.

## 9.2 문서와 개체를 분리한다

```text
문서(Document) = 특정 사건을 설명하는 원고 또는 BOOK 단락
개체(Entity)  = 인물·장소·개념·사건 자체
```

예를 들어 `02-birth.md`는 문서이고, `maya-devi`, `lumbini`, `event-buddha-birth-at-lumbini`는 개체다.

## 9.3 처음부터 거대한 Graph DB를 만들지 않는다

초기 OL에서는 Neo4j, RDF, SPARQL 같은 무거운 시스템보다 다음 구조가 적합하다.

```text
Markdown
+ YAML frontmatter
+ Entity Registry
+ Relation triples
+ JSON export
+ Astro build aggregation
```

이 구조는 Git 친화적이고, AI 처리에 적합하며, 단일 HTML 철학과도 충돌하지 않는다.

# 10. OL BOOK과 Index Layer 원칙

OL BOOK은 독립 수행 객체다. 즉 한 권의 BOOK은 단일 HTML로 보관·공유·오프라인 열람이 가능해야 한다.

하지만 전체 문헌 간 의미망은 OL HOME이 담당한다.

```text
OL BOOK = 독립 실행 가능한 한 권의 수행 객체
OL HOME = 전체 BOOK과 WORKS와 DESIGN과 Entity를 연결하는 Index Layer
```

따라서 Canonical Spine은 두 가지 출력을 모두 염두에 두어야 한다.

```text
1. book internal data
   - BOOK 안에서 읽기 위한 카드·문서·목차·내부 링크

2. semantic export
   - OL HOME이 수집할 entities / relations / source map
```

장기적으로 권장되는 출력:

```text
public/books/buddha-story/index.html
public/books/buddha-story/buddha-story.meta.json   ← 장기 확장안
src/content/book/buddha-story.md
```

`meta.json` 예시:

```json
{
  "book_id": "buddha-story",
  "spine_version": "v1.0",
  "entities": [
    { "id": "maya-devi", "type": "person" },
    { "id": "lumbini", "type": "place" },
    { "id": "event-buddha-birth-at-lumbini", "type": "event" }
  ],
  "relations": [
    {
      "subject": "maya-devi",
      "predicate": "gave_birth_to",
      "object": "siddhartha-gautama"
    },
    {
      "subject": "siddhartha-gautama",
      "predicate": "born_in",
      "object": "lumbini"
    }
  ]
}
```

# 12.2  `00-canonical-spine/` 안에 두는 10개 산출물

 모두 **Canonical Spine 작업의 핵심 관리 파일**입니다. 역할을 간단히 말하면, `spine-v0.1.yaml`이 “기계가 읽는 기준 데이터”이고, 나머지 파일들은 그것을 사람이 검토·보정·제작·출판·지식그래프화하기 위한 보조 인덱스입니다. 업데이트본에서는 Canonical Spine을 원전 비교·번역뿐 아니라 WORKS, BOOK, DESIGN, Entity, Ontology, Knowledge Graph 확장의 기준으로 삼는다고 정의합니다.

|파일|역할과 기능|
|---|---|
|`1. spine-v0.1.yaml`|**Canonical Spine의 원본 데이터베이스**입니다. Episode, Sequence, Event ID, 원전 대응 여부, 주요 인물·장소·개념, relation 후보, design 필요 항목, BOOK 내부 anchor 등을 구조화해서 담습니다. AI 에이전트와 후속 자동화가 읽을 “주 파일”입니다.|
|`2. spine-v0.1.md`|`spine-v0.1.yaml`을 사람이 읽기 좋게 풀어쓴 **검토용 문서**입니다. 전체 목차, 사건 흐름, 장별 구성, 편집 의도를 사람이 빠르게 확인할 수 있습니다.|
|`3. event-index.csv`|모든 Event를 한 줄씩 정리한 **사건 색인표**입니다. `event_id`, 제목, 순서, 소속 Episode/Sequence, 상태 등을 표로 관리합니다. 빠른 정렬·검색·누락 확인에 유용합니다.|
|`4. source-coverage-matrix.csv`|각 Event가 4종 원전에서 어떻게 나타나는지 표시하는 **원전 대응표**입니다. `present`, `partial`, `expanded`, `absent`, `uncertain` 등을 사용하여 어느 원전이 어떤 사건을 다루는지 한눈에 봅니다.|
|`5. entity-candidates.yaml`|Canonical Spine에서 추출된 **Entity 후보 목록**입니다. 인물, 장소, 개념, 사건, 상징 등을 `src/content/entities/`에 등록하기 전 초안으로 모읍니다. OL 구조에서는 `entities` 컬렉션이 인물·장소·개념 등을 담당하므로 이 파일이 Entity Registry의 씨앗이 됩니다.|
|`6. relation-candidates.yaml`|사건 속 의미 관계를 triple 형식으로 정리한 **Knowledge Graph 관계 후보집**입니다. 예: `maya-devi → gave_birth_to → siddhartha-gautama`. 태그와 달리 “누가-무엇을-어떻게 연결되는가”를 데이터화합니다.|
|`7. design-needs.csv`|각 Event에 필요한 삽화·캐릭터·배경·소품 레퍼런스를 정리한 **시각 자산 제작 요청표**입니다. 예: 룸비니 탄생 장면 삽화, 마야부인 캐릭터 시트, 룸비니 배경 레퍼런스 등. DESIGN은 이미지 파일과 메타데이터를 분리해 관리하므로, 이 파일은 향후 `public/design/`과 `src/content/design/` 등록의 사전 목록이 됩니다.|
|`8. works-book-map.yaml`|제작 중 원고인 WORKS와 완성 출판물인 BOOK을 연결하는 **출판 경로 매핑 파일**입니다. 어떤 Episode/Sequence가 `src/content/works/buddha-story/*.md`로 가고, 최종적으로 `public/books/buddha-story/index.html` 및 `src/content/book/buddha-story.md`와 어떻게 연결되는지 기록합니다. WORKS는 초고·번역·주석 작업장이고 BOOK은 완성 HTML 출판물입니다.|
|`9. needs-human-review.md`|AI가 판단하기 어렵거나 인간 감수가 필요한 항목을 모으는 **검토 요청 목록**입니다. 원전 대응이 불확실한 사건, 신화적 장면 처리, 사건 분할 여부, 번역 용어 선택, 한역과 GCB의 차이 등을 기록합니다.|
|`10. editorial-decisions.md`|사람이 최종적으로 내린 편집 판단을 기록하는 **편집 결정 로그**입니다. 예: “과거생 서사는 프롤로그로 둔다”, “칠보행은 상징적 서술로 보존한다”, “GCB 기준 순서를 유지하되 한역 확장 묘사는 주석층에 반영한다” 같은 결정이 여기에 축적됩니다.|

정리하면:

```text
spine-v0.1.yaml              = 전체 구조의 원본 데이터
spine-v0.1.md                = 사람이 읽는 목차·해설본
event-index.csv              = 사건 목록 색인
source-coverage-matrix.csv   = 4종 원전 대응표
entity-candidates.yaml       = Entity 등록 후보
relation-candidates.yaml     = Knowledge Graph 관계 후보
design-needs.csv             = 삽화·레퍼런스 제작 목록
works-book-map.yaml          = WORKS → BOOK 출판 연결표
needs-human-review.md        = 인간 검토 필요 목록
editorial-decisions.md       = 최종 편집 판단 기록
```

이 10개를 함께 운용하면 Canonical Spine은 단순 목차가 아니라 **원전 정리, 번역, 합본, 삽화 제작, OL HOME 등록, BOOK 출판, 지식그래프 확장까지 이어지는 통합 제작 좌표계**가 됩니다.

# 15. 핵심 결론 — 업데이트판

Canonical Spine 단계에서 AI가 해야 할 일은 다음이다.

```text
목차 추출
사건 후보 분해
원전 대응표 생성
불확실 항목 보고
Entity 후보 추출
Relation 후보 추출
DESIGN 필요 목록 생성
WORKS/BOOK 대응표 생성
```

당신이 해야 할 일은 다음이다.

```text
작품 Premise 결정
사건 단위 승인
GCB와 한역 전승 사이의 편집 균형 판단
신화적 장면 처리 원칙 결정
용어와 사상적 방향성 결정
Entity와 relation의 최종 의미 판단
최종 독서 경험의 호흡과 문체 결정
```

이 단계의 가장 중요한 원칙은 다음 한 문장이다.

> Canonical Spine은 “무엇을 쓸 것인가”를 정하는 단계가 아니라, “원전·번역·삽화·출판·의미망을 어디에 붙여서 함께 운영할 것인가”를 정하는 단계다.

따라서 이 단계에서는 문장을 아름답게 만들려고 하지 말고, **구조를 튼튼하게 만드는 것**이 최우선이다. 이제 그 구조는 원고 내부의 목차를 넘어, OL HOME 전체의 WORKS·BOOK·DESIGN·Entity·Ontology·Knowledge Graph를 잇는 첫 번째 뼈대가 되어야 한다.
