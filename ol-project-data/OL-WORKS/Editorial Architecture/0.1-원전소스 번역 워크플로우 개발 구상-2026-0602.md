# 원전소스 번역 워크플로우 개발 구상-편집 설계(Editorial Architecture)구상
# 질의

불교스토리라는 부처님 일대기를 정리하는 프로젝트를 시작하려고 해. 이 프로젝트는 1차 텍스트 중심의 원고탈고. 2차 삽화추가로 이어질 예정. 1차 원고탈고를 위해 현재 4종의 원전 소스를 준비함. 2종은 영어원고(The Great Chronicle of Buddhas), 2종은 한문원고-한국어번역본(불본행집경, 불본행경)이 병기되어 있으나 추가 수정작업 필요. 2종 영어원고는 원작품과 개정판. 나는 이 원전소스를 최종 원고로 탈고하기 까지 로컬 AI 에이전트 워크플로우를 기획 중이야. 우선 내가 구상한 과정은 아래와 같음. 이에 대한 ai 기획 개발 및 워크플로우 전문가이면서 불교문헌 번역 전문가로서 추가해야 하거나 수정 및 보완해야 할 사항에 대해 의견을 제안해줘.

1. 이 4종 원고 각각을 단일 목차 체계(The Great Chronicle of Buddhas 목차를 기준으로 설정)로 분류하여 관련된 주제 중심으로 불본행집경, 불본행경의 내용을 모으는 작업. 새로운 폴더를 목차를 기준으로 만들어서 새롭게 원전소스 분류하는 작업.
2. 단일 목차 각 소주제로 묶인 원전소스 4종에 대한 한국어 일상어 1차번역. 번역 진행하면서 번역용어집(원전 언어:한국어 병기하고 이 용어가 사용된 원전소스 출처기록) 제작- 이 번역용어집은 원전소스 번역의 기준이 됨.
3. 내가 용어집을 검토하고 최종적으로 사용할 번역 용어를 선택. 이 선택된 용어를 가지고 2차 번역작업 진행.
4. 2차 번역된 4종 원고를 상호 비교하고 모든 내용을 빠짐없이 하나의 원고로 합치하는 작업. 하나로 합치하되 문맥과 글의 분위기를 유지하고 글의 흐름이 매끄럽게 세부 조정. 
5. 내가 하나로 합치된 원고를 검토하고 세부 조정. 내가 직접 수정하거나 양이 많으면 수정 지시함.

이렇게 하나의 워크플로우 싸이클이 마무리되면 4종 원고 각각에 대한 한국어 번역본, 번역용어집, 최종 한국어 합본이 만들어 짐. 이를 OL 홈페이지 WORKS 페이지에서 업로드함. 이렇게 업로드 하면서 모든 원전소스에 대한 번역 싸이클이 마무리되면 'OL BOOK - 붓다스토리'로 홈페이지 출판.


---

# 응답

현재 구상은 상당히 정교하며, 특히 다음 세 가지 점이 매우 뛰어납니다.

1. “원전별 번역”과 “최종 서사 통합”을 분리했다는 점
2. “용어집(Terminology Base)”을 초기에 구축하려는 점
3. 최종 목표를 단순 번역이 아니라 “OL BOOK용 현대 한국어 붓다 서사”로 명확히 설정했다는 점

이 방식은 단순 번역 프로젝트가 아니라, 현대 불교 디지털 출판에서 말하는:

* Critical Edition Workflow
* Translation Memory Workflow
* Narrative Harmonization Workflow
* Canonical Knowledge Pipeline

을 동시에 수행하는 구조에 가깝습니다.

특히 당신이 선택한:

> “The Great Chronicle of Buddhas를 기준 목차(canonical spine)로 삼고, 다른 원전을 그 spine에 매핑”

하는 방식은 매우 좋은 선택입니다.

왜냐하면 The Great Chronicle of Buddhas 는:

* 현대 독자 친화적 서사 구조
* 사건 흐름의 연속성
* 비교적 안정적인 chronology
* 부처님 생애 전체의 narrative continuity

를 이미 갖추고 있기 때문입니다.

반면 불본행집경 과 불본행경 은:

* 사건 중복
* 삽입 설화
* 장 구조 불균형
* 상이한 편집 전통

이 존재하므로 “기준 spine 없이” 병합하면 매우 혼란스러워질 가능성이 큽니다.

다만 현재 워크플로우에는 몇 가지 중요한 층위가 추가될 필요가 있습니다.

# 전체적으로 추천하는 개선 구조

현재 구조:

```text
목차 정리
→ 1차 번역
→ 용어집 구축
→ 2차 번역
→ 합본
→ 검수
```

권장 구조:

```text
[0] Canonical Spine 구축
→
[1] 원전 구조 분해 및 segment 단위화
→
[2] 사건 ID(Event ID) 부여
→
[3] 번역 메모리 + 용어집 구축
→
[4] 1차 직역층 생성
→
[5] 의미 기반 2차 번역
→
[6] 원전 비교·충돌 분석
→
[7] Narrative Merge
→
[8] 현대 한국어 literary polishing
→
[9] Reference Layer 생성
→
[10] OL BOOK 최종 패키징
```

이 중 핵심적으로 추가해야 하는 것은 아래 다섯 가지입니다.

---

# 1. “문단”이 아니라 Segment 단위로 잘라야 함 (매우 중요)

현재 가장 중요한 구조적 보완점입니다.

당신은 지금:

> “목차별 폴더”

를 생각하고 있는데, 이것만으로는 부족합니다.

왜냐하면 불교 원전은:

* 같은 사건이 다른 길이로 반복되고
* 문단 경계가 판본마다 다르며
* 삽입 게송이 존재하고
* 동일 사건이 부분적으로 흩어져 있기 때문입니다.

따라서 AI 워크플로우에서는:

```text
Chapter
  └── Event
        └── Segment
```

구조가 필요합니다.

예:

```text
01-bodhisatta-descent/
  event-001-tuṣita-decision/
    seg-001-gcb-en.md
    seg-002-gcb-revised-en.md
    seg-003-불본행집경.md
    seg-004-불본행경.md
```

이 구조가 필요한 이유:

* AI context contamination 감소
* 비교 정확도 증가
* hallucination 감소
* parallel translation 가능
* diff 추적 가능

입니다.

이 단계는 사실상:

> “문헌을 AI 처리 가능한 atomic unit로 분해”

하는 작업입니다.

OL 철학의 “원자성 + 관계성”과도 일치합니다. 

---

# 2. Event ID 체계를 반드시 먼저 설계해야 함

이것은 향후 OL BOOK 전체 ontology의 핵심이 됩니다.

예:

```text
BL-001 Birth prophecy
BL-002 Queen Maya dream
BL-003 Lumbini birth
BL-004 Asita prophecy
```

추천 규칙:

```text
BL = Buddha Life
EV = Event
SC = Scene
```

예:

```text
BL-EV-004-asita-prophecy
```

이 구조가 왜 중요하냐면 이후:

* 삽화 연결
* 지도 연결
* 등장인물 연결
* 웹툰 컷 연결
* AI retrieval
* semantic search
* citation
* 다국어 연결

모든 것이 Event ID 중심으로 연결되기 때문입니다.

즉:

> 지금 만드는 Event ID가 미래 OL ontology의 씨앗

입니다.

---

# 3. “번역”을 최소 3개 층으로 나눠야 함

현재 구상은:

* 1차 번역
* 2차 번역

인데 실제로는 최소 3층이 필요합니다.

권장:

| 층               | 목적            |
| --------------- | ------------- |
| Literal Layer   | 원문 충실 직역      |
| Semantic Layer  | 의미 기반 자연어 변환  |
| Narrative Layer | 최종 독서용 문학적 흐름 |

---

## 3-1. Literal Layer

예:

```text
보살은 도솔천에서 관찰하였다
```

이 층은:

* 원문 대응 추적
* 역검증
* AI hallucination 감시

용도입니다.

절대 버리면 안 됩니다.

---

## 3-2. Semantic Layer

일반 독자 이해용.

예:

```text
보살은 인간 세상에 태어날 시기를 살폈다.
```

---

## 3-3. Narrative Layer

OL BOOK 최종본.

예:

```text
그는 긴 침묵 속에서 인간 세상을 바라보았다.
이제 다시 그곳으로 내려갈 때가 왔다.
```

이 3층을 분리하지 않으면 나중에:

* 원문 검증
* 학술성
* 대중성
* 문학성

이 서로 충돌합니다.

---

# 4. 용어집은 “사전(dictionary)”이 아니라 TBX 구조로 가야 함

현재 구상은 매우 좋지만 더 발전시켜야 합니다.

권장 필드:

```yaml
term_id:
source_lang:
target_lang:
literal_translation:
preferred_translation:
alternative_translation:
usage_note:
do_not_use:
source_reference:
first_appearance:
related_terms:
```

예:

```yaml
term_id: TERM-00034
source_lang: bodhisatta
preferred_translation: 보살
alternative_translation:
  - 보리살타
  - 깨달음을 구하는 이
usage_note:
  초기불교 문맥에서는 미래불 의미 강조
source_reference:
  - GCB Vol.1 p.33
  - 불본행집경 권3
```

이 구조는 미래:

* AI retrieval
* 자동 용어 통일
* glossary export
* multilingual support

에 필수입니다.

---

# 5. “원전 충돌 관리 레이어”가 필요함

이건 매우 중요합니다.

불교 전기 문헌은:

* 사건 순서 차이
* 초자연 요소 차이
* 대승적 확장
* 후대 삽입

이 존재합니다.

예:

* 마야부인의 꿈 해석
* 탄생 직후 행보
* 출가 동기
* 마왕 대결

등은 판본마다 차이가 큽니다.

따라서 반드시:

```text
variant-notes/
```

레이어를 따로 만들어야 합니다.

예:

```yaml
event: BL-EV-003
variant:
  source: 불본행집경
  difference:
    "탄생 직후 칠보행 묘사 추가"
editorial_decision:
  "최종본에는 축약 반영"
```

이게 없으면 후반에:

* 왜 삭제했는지
* 왜 선택했는지
* 어디서 왔는지

추적이 불가능해집니다.

---

# 6. AI 워크플로우 관점에서 가장 중요한 추가 제안

당신은 “로컬 AI 에이전트 워크플로우”를 말했는데, 여기서 핵심은:

> “한 번에 긴 문서를 처리하지 않는 것”

입니다.

추천 방식:

```text
Segment 단위 처리
→
Event 단위 merge
→
Chapter 단위 polishing
→
Book 단위 consistency review
```

즉:

* 번역 에이전트
* 용어검사 에이전트
* 판본비교 에이전트
* merge 에이전트
* literary polishing 에이전트

를 분리해야 합니다.

하나의 거대 프롬프트로 해결하려 하면:

* 문체 붕괴
* 기억 오염
* 용어 불안정
* hallucination

이 급증합니다.

---

# 마지막으로 — 가장 중요한 전략적 조언

당신 프로젝트는 단순 “불교 번역”이 아니라 사실상:

> “현대 한국어 붓다 생애 canonical reconstruction”

입니다.

즉:

* 번역
* 편집
* 문헌비교
* 대중화
* 디지털 출판
* AI 협업

이 모두 결합된 프로젝트입니다.

따라서 가장 중요한 것은:

> “처음부터 구조를 남기는 것”

입니다.

AI 시대에는:

* 원문보다
* 구조화된 중간 산출물

이 더 중요해질 가능성이 큽니다.

당신이 만드는:

* Event map
* terminology base
* variant notes
* aligned segments

자체가 향후 OL AI의 핵심 학습 자산이 될 수 있습니다.
