문서뷰 업그레이드 로드맵

요구사항 최종 확정:

	•	편집기: 단일 편집창, 상단 툴바 + / 슬래시 커맨드 팝업 (분할 프리뷰 아님)
	•	문서뷰: 우상단 “편집” 버튼 → 같은 화면에 툴바 등장하여 인라인 편집
	•	이전/이후: 컬럼 순 → 카드 순 1차원 순회
	•	모드 전환 경고 표시
	•	카드 메타(태그/우선순위/컬럼) 문서뷰 상단 표시

데이터 구조 변경

카드 스키마 (기존 → 신규)

// 기존
{ id, title, body, tags, priority, status, ... }

// 신규
{
  id, title,
  body,             // 그대로 유지 (plain 모드용 - 하위 호환)
  bodyMode,         // 'plain' (기본) | 'markdown'
  bodyMd,           // 마크다운 모드일 때 사용 (없으면 body 폴백)
  tags, priority, status, ...
}


왜 body와 bodyMd를 분리하나

	•	기존 카드는 bodyMode 미정의 → 'plain' 으로 간주, 기존 body 그대로 사용
	•	마크다운 모드 켜도 body(plain)는 백업으로 남아있음 → 모드 토글 시 데이터 손실 없음
	•	단, 동기화 정책: plain 모드에서 편집한 내용은 body 갱신 / markdown 모드에서 편집한 내용은 bodyMd 갱신, 양방향 자동 변환은 안 함 (마크다운 → plain은 정보 손실되므로)

Phase 분할

## 📦 Phase 1 — 마크다운 파서 (기반)

목표: 외부 의존성 없이 마크다운 → HTML 변환기 자체 구현

구현 범위

	•	블록 파싱: 제목 (#~###), 단락, 인용 (>), 코드 블록 (```), 구분선 (—), 순서/비순서 리스트, 체크박스
	•	인라인 파싱: **굵게**, *기울임*, ~~취소선~~, `코드`, [링크](url), ![alt](src)
	•	XSS 방어: <script>, on*= 속성, javascript: URL 차단
	•	라이브러리 없이 정규식 + 라인 단위 스캐너로 ~150줄 추정

검증 포인트

	•	기존 plain text도 마크다운 파서에 그대로 통과시켜도 깨지지 않음 (단순 단락으로 처리)
	•	한국어 마크다운(공백, 띄어쓰기) 정상 처리
	•	base64 data URL 이미지 정상 렌더링

산출물: parseMarkdown(md) 함수 + escapeHTML() + 단위 테스트 케이스 5~6개

## 📦 Phase 2 — 카드 데이터 구조 확장 + 모달 모드 토글

목표: 기존 카드 모달에 “마크다운 편집 전환” 버튼 추가 (편집기 자체는 다음 단계)

구현 범위

	•	카드 객체에 bodyMode, bodyMd 필드 추가 (마이그레이션: 기존 카드는 bodyMode='plain' 자동 부여)
	•	카드 모달 헤더 우측에 두 버튼 추가:
	•	🔀 모드 토글: plain ↔ markdown 전환 (전환 시 confirm 토스트로 경고)
	•	📖 문서뷰로 이동 (이건 Phase 4에서 활성화)
	•	모드별로 본문 영역만 다르게 렌더 (plain은 기존 textarea, markdown은 일단 더 큰 textarea로 시작)
	•	addHistory('mode_changed', ...) 기록 추가 — 누가 언제 마크다운 전환했는지 추적

검증 포인트

	•	기존 카드 정상 로드 (bodyMode 없어도 plain으로 동작)
	•	모드 전환 후 저장 → 재진입 시 모드 유지
	•	변경이력 페이지에 모드 전환 기록 표시

산출물: 카드 모달 본문 영역 분기 렌더, 모드 토글 핸들러

## 📦 Phase 3 — 마크다운 에디터 (툴바 + 슬래시 커맨드)

목표: 노션 스타일 인라인 편집기 구현

3-A. 툴바

	•	모달 본문 영역 상단에 sticky 툴바
	•	아이콘 버튼: H1 / H2 / H3 / B / I / S / </> / 인용 / 불릿 / 번호 / 체크 / 링크 / 이미지 / 구분선
	•	동작: 선택 영역 또는 현재 줄에 마크다운 문법 삽입/래핑
	•	lucide-style SVG 아이콘 (디자인 정책 준수)

3-B. 슬래시 커맨드

	•	빈 줄에서 / 입력 → 캐럿 위치에 팝업 띄움
	•	팝업: 위와 동일한 블록 타입 목록 + 키보드 ↑↓ 네비
	•	Enter / 클릭 시 해당 줄을 선택한 블록 타입으로 변환 (/ 문자는 제거)
	•	Esc로 팝업 닫기
	•	팝업 내 검색 (예: /h2, /이미지)

3-C. 이미지 삽입 다이얼로그

	•	이미지 버튼 또는 슬래시 → 이미지 클릭 시 작은 서브 모달
	•	두 탭: URL 입력 / 파일 업로드
	•	파일 업로드: <input type="file"> → FileReader → base64 변환 → 크기 경고 (>500KB 시 노란 경고, >2MB 시 빨간 경고)
	•	삽입 후 ![alt](url 또는 data:...) 텍스트로 본문에 추가

3-D. 링크 삽입 다이얼로그

	•	텍스트 선택 + 링크 버튼 → 작은 popover
	•	URL 입력 받음 → [선택텍스트](url) 로 래핑
	•	javascript: 시작 URL 거부

검증 포인트

	•	키보드 네비 (Tab으로 툴바 진입, ↑↓로 슬래시 메뉴, Esc 닫기)
	•	마크다운 문법이 올바르게 삽입됨
	•	큰 base64 이미지(>2MB) 경고 후 사용자가 확정해야 삽입
	•	모바일/터치에서도 툴바 사용 가능

산출물: ~400~500줄 추정. 가장 큰 단위.

## 📦 Phase 4 — 문서뷰 (Document View)

목표: 가독성 중심 읽기 뷰 + 카드 순회

4-A. 뷰 라우팅 + 헤더 메뉴

	•	사이드바/헤더에 “문서뷰” 항목 추가
	•	currentView = 'document' 상태 추가
	•	문서뷰 진입 시 currentDocCardId 별도 상태로 관리
	•	헤더 메뉴 클릭 시: 칸반 첫 컬럼의 첫 카드를 자동 선택

4-B. 카드 순회 헬퍼

	•	getOrderedCardList() → 컬럼 순서 → 컬럼 내 카드 순서로 1차원 배열 반환
	•	getPrevNextCard(cardId) → 이전/이후 카드 ID 반환

4-C. 문서뷰 레이아웃

	•	최대 폭 ~720px, 좌우 자동 마진
	•	본문 폰트 크기 17~18px, line-height 1.7
	•	상단: 카드 제목, 메타 정보 (컬럼명, 태그 칩, 우선순위, 학습상태)
	•	본문: parseMarkdown(card.bodyMd || card.body) 결과 렌더
	•	우상단 액션 바:
	•	편집 버튼 (이 페이지 내 인라인 편집 활성화)
	•	모달로 열기 (전체 편집 모달)
	•	컬럼/그리드 뷰로 돌아가기
	•	하단:
	•	← 이전: [카드 제목] / 다음: [카드 제목] → 링크
	•	첫/마지막이면 disabled

4-D. 카드 우상단 뷰어 아이콘

	•	칸반 카드 / 그리드 카드 / 리스트 행 우상단에 작은 📖 아이콘 추가
	•	클릭 시 → 해당 카드로 문서뷰 진입 (이벤트 전파 차단해서 모달 안 열리게)
	•	카드 모달 헤더에도 동일 아이콘 추가

검증 포인트

	•	첫/마지막 카드에서 prev/next 버튼 적절히 비활성화
	•	빈 칸반 (카드 없음) 상태에서 문서뷰 빈 상태 표시
	•	카드 삭제 후 문서뷰 진입 시 안전 처리

산출물: 신규 뷰 1개 + 헬퍼 2개 + 아이콘 통합. ~300줄 추정.

## 📦 Phase 5 — 인라인 편집 모드 (문서뷰 내)

목표: 문서뷰의 “편집” 버튼 → 같은 화면에서 Phase 3 에디터 활성화

구현 범위

	•	문서뷰에 editing: true/false 상태 추가
	•	편집 모드 시:
	•	본문 영역의 렌더된 HTML이 → Phase 3의 마크다운 에디터로 교체
	•	상단 툴바 등장 (모달과 동일한 컴포넌트 재사용)
	•	우상단 “저장” / “취소” 버튼
	•	저장 시 bodyMd 갱신 + addHistory('edited', ...) + 토스트
	•	취소 시 원래 내용 복원

검증 포인트

	•	편집 모드 중 prev/next 버튼 클릭 시 → 변경사항 있으면 confirm
	•	저장 후에도 같은 카드가 문서뷰에 머무름 (페이지 이동 안 함)
	•	Phase 3 에디터를 모달과 문서뷰 양쪽에서 재사용 가능한 형태로 추출되어 있어야 함 (Phase 3에서 컴포넌트 분리 의식해서 작성)

산출물: ~150줄 추정 (Phase 3 컴포넌트 재사용 가정).

## 📦 Phase 6 — 마무리 / 폴리싱

	•	마크다운 인쇄 스타일 (@media print) — 배포용 PDF 출력 대비
	•	변경이력에 mode_changed, 인라인 편집 액션 통합
	•	빈 상태 일러스트 추가 (문서뷰 카드 없음 등)
	•	키보드 단축키 (선택):
	•	Ctrl+B, Ctrl+I → 굵게/기울임
	•	Ctrl+E → 문서뷰에서 편집 진입
	•	[, ] → 문서뷰에서 이전/다음 카드

작업 순서 권장 사항

Phase 1 (파서)         ─┐
Phase 2 (스키마+토글)   ├─ 데이터·기반
Phase 3 (에디터)       ─┤
Phase 4 (문서뷰)       ─┤
Phase 5 (인라인 편집)  ─┘ — 마무리·통합
Phase 6 (폴리싱)


각 Phase별 검증 후 확정 → 다음 Phase 진행 (기존 작업 패턴 유지)

사전 결정 사항 (Phase 1 시작 전)

	1.	마크다운 파서를 자체 구현 vs 인라인 미니 라이브러리 임베드
	•	자체 구현 권장 (단일 HTML 정책 부합, ~150줄로 충분)
	•	단, marked.js 같은 작은 파서를 인라인으로 박는 것도 가능
	2.	bodyMd 필드 추가 시 기존 export 호환성
	•	신규 필드는 누락된 카드에서 그대로 무시되도록 설계 → 하위 호환 보장
	3.	이미지 base64 크기 임계값
	•	경고 임계: 500KB / 거부 임계: 2MB 정도 권장 (단일 HTML 파일 비대화 방지)
	•	또는 사용자가 임계값을 못 거부하게 하지 말고 경고만 띄우고 결정 위임
	4.	슬래시 커맨드 트리거 규칙
	•	빈 줄에서만 vs 어디서든 작동
	•	노션은 어디서든 작동하나, 빈 줄만 허용이 구현 단순함. 일단 빈 줄에서 시작 권장.
