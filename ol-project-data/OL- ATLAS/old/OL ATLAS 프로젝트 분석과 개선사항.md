# OL ATLAS 프로젝트 분석과 개선사항
  
# 개요

  OL ATLAS는 단일 HTML 파일로 동작하는 콘텐츠 에디터 겸 뷰어입니다. 서버/계정 없이 브라우저만으로 카드 기반 콘텐츠를 작성·관리·배포할 수 있도록 설계되었습니다. "한
  파일 = 에디터 + 뷰어 + 데이터"라는 독특한 컨셉이 핵심입니다.

  - 버전: 0.0.7 (schemaVersion: 10)
  - 빌드 도구: esbuild (IIFE 번들)
  - 프레임워크: 없음 (Vanilla JS)
  - 의존성: devDependencies에 esbuild 하나뿐

  ---
  아키텍처

  빌드 파이프라인

  src/main.js → esbuild (IIFE bundle) → dist/bundle.js
                                            ↓
  src/index.html + CSS + bundle.js → inline.mjs → dist/ol-atlas_v0.0.7.html (자기완결형)

  build.mjs가 JS를 번들링하고, inline.mjs가 HTML 셸의 플레이스홀더(<!--STYLES-->, <!--SCRIPTS--> 등)를 실제 CSS/JS/favicon으로 치환하여 모든 것이 포함된 단일 HTML 
  파일을 생성합니다.

  상태 관리 (Redux-like 패턴)

  - store.js: 중앙 상태 저장소 (storeInit, getState, applyState), 뷰 구독 관리
  - state.js: S Proxy 객체 — 읽기만 허용, 직접 set하면 에러 발생 (불변성 강제)
  - action.js: dispatch(action) → reducer 체인 → applyState → markDirty → queueRender (Flux 패턴)
  - render-queue.js: 렌더링 요청을 큐잉/배치 처리

  데이터 흐름

  dispatch({ type, payload }) → reducers → applyState → markDirty → queueRender → 뷰 갱신

  데이터 저장

  - localStorage 기반 로컬 저장 (서버 통신 없음)
  - 내보내기 시 __LOADED_DATA_B64__에 Base64로 데이터를 삽입하여 HTML 파일 자체가 데이터를 품음
  - 다시 열면 임베디드 데이터를 자동 로드

  주요 설계 특징
  
  1. 프레임워크 없는 Vanilla JS — React/Vue 없이 Proxy 기반 상태 관리 + subscribe 패턴으로 구현
  2. 자기완결형 단일 HTML — 빌드 결과물이 CSS/JS/데이터/이미지를 모두 품은 하나의 .html 파일
  3. Flux/Redux 패턴 — dispatch → reducer → applyState → queueRender 흐름이 상태 변경의 유일한 경로
  4. 해시 라우팅 — #kanban, #document/slug 형식으로 뷰 전환, hashchange 리스닝
  5. 스키마 마이그레이션 — schema.js로 데이터 구조 버전 업그레이드 자동 처리
  6. 다중 내보내기 — HTML(재배포 가능), JSON, Markdown(단일/개별 파일) 지원
  7. 에디터 추적 — 카드별 acts 배열에 편집자별 수정 이력 기록

✻ Baked for 1m 6s

# 개선할 영역 점검

❯ 현재 유지보수 효율성을 위해서 코드를 분리했는데 js, css가 각 기능과 페이지들에 맞게 분산되어 있는 것인가? 웹개발 전문가로서 현재 잘 분산되어 있는지 점검하고      
  어뚱한 곳에 코드들이 배치되어 있지는 않은지 점검해줘. 직접 코드 수정은 하지 말고 우선 분석만해줘.                                                                
---
  코드 배치 점검 결과
  
  전체적으로 보면 디렉토리 구조 설계 자체는 합리적입니다. core/, actions/, components/, ui/, data/ 분리가 잘 되어 있고, Flux 패턴의 흐름(action → reducer →
  render)이 명확합니다. 다만 실제 코드 내용을 보면 여러 곳에서 배치가 어긋나 있는 문제들이 있습니다.

## 2. JS: about.js에 휴지통 로직이 들어있음 (497줄)
  
  components/shared/about.js라는 파일에 About 뷰 + 휴지통 뷰가 합쳐져 있습니다.

  - renderTrash(), initTrashHandlers(), _trashRestoreCard(), _trashDelCard() 등 휴지통 전용 함수가 ~140줄
  - subscribe('trash', renderTrash) — 별도 뷰로 등록까지 되어 있음
  - buildAboutTrashSection() — 사이드바에 About+휴지통 링크를 그림

  About과 Trash는 별개의 뷰인데 한 파일에 있어서, 나중에 휴지통만 수정하려면 About 코드를 함께 건드려야 합니다.

## 3. JS: events.js가 중앙 집중형 이벤트 와이어링 (503줄)
  
  설계 의도는 이해하지만, 실제로는 각 기능의 내부 동작까지 events.js에 들어있습니다:

  - 리스트뷰 컬럼 설정 팝오버 (62~188줄) — DOM 생성·드래그 정렬까지 포함. listview.js에 있어야 함
  - 모바일 검색 전체 로직 (395~466줄) — search.js에 있어야 함
  - 모바일 사이드바 저장/열기 버튼 연결 (468~503줄) — export-import.js에 있어야 함

  단순한 addEventListener 연결은 중앙 파일에 있어도 괜찮지만, 60줄짜리 드래그 핸들러, 40줄짜리 모바일 검색 UI는 해당 컴포넌트에 있어야 합니다.

## 4. JS: tag-filter.js가 core/에 있지만 DOM을 직접 조작함
  
  core/ 디렉토리는 순수 로직(상태, 라우팅, 유틸 등)이 있는 곳인데, tag-filter.js는:

  - document.getElementById('tag-filter-btn').addEventListener(...) (86줄)
  - document.getElementById('tfd-clear-btn').addEventListener(...) (99줄)
  - document.addEventListener('click', ...) (107줄)

  DOM 이벤트 리스너를 직접 등록하고 있어서, 성격상 components/ 또는 events.js에 있어야 합니다. 같은 파일 안에 _currentEditingCard라는 docview 편집 상태까지 관리하고
  있는데, 이것은 태그 필터와 무관한 상태입니다.

## 5. JS: constants.js에 이미지 처리 로직이 포함됨
  
  core/constants.js(135줄)에 다음이 들어있습니다:

  - bodyImagesToTokens() — 마크다운 본문의 이미지를 토큰으로 변환
  - bodyTokensToStandardMd() — 토큰을 표준 마크다운으로 복원
  - newImgId(), safeImgAlt() — 이미지 ID/alt 유틸

  이것들은 상수가 아니라 이미지 처리 유틸입니다. core/image-utils.js 또는 core/body-helpers.js(현재 12줄뿐)에 있어야 합니다.

## 6. JS: export-import.js가 actions/에 있지만 순수 액션이 아님
  
  actions/export-import.js(601줄)은:

  - 순수 ZIP 생성기 (94~189줄, ~100줄)
  - 마크다운 프런트매터 파서 (421~483줄, ~60줄)
  - 드롭다운 UI 열기/닫기 (24~60줄)
  - DOM 이벤트 리스너 다수 (document.getElementById('export-btn').addEventListener(...) 등)

  actions/ 폴더의 다른 파일들(card-actions.js, column-actions.js)은 순수하게 action creator + reducer만 담고 있는데, export-import.js만 UI 로직 + 파일 변환 로직 + 
  DOM 이벤트가 전부 섞여 있습니다.
