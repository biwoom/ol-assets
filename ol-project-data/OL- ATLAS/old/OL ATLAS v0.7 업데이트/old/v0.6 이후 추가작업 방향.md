# v0.6 이후 추가작업 방향

# 내 질문1
[내 질문]
현재 OL ATLAS v0.5를 v0.6으로 업그레이드 하기 위해 첨부한 작업과정을 진행했어. 개발과 배포를 분리하기위해 코드분리하고 가상서버 기능도 추가했음. 당신이 보기에 이후 더 진행해야할 부분이 있다면 제안해줘. 나는 차후 티들리위키처럼 OL 에 설정페이지 추가후 특정 플러그인 모듈을 사용자가 추가하게 하고 싶어. 그리고 브라우저 새로고침시 업데이트한 내용 유실되지 않게 경고창도 만들어야 할 것 같아. 나는 OL 코드가 더 모듈화되고 버그가 최소화 되고 유지보수가 편리했으면 좋겠어. 이런 방향에서 추가 작업과제들을 제안해줘.

---

# 전문가 제안서 1

현재 당신의 v0.6 작업 방향은 상당히 올바릅니다. 특히 다음 4가지는 장기 유지보수 관점에서 매우 좋은 결정입니다.

* build.js 기반 모듈 분리
* `src/ui`, `src/core` 구조화
* custom-select 공통화
* portal 기반 overflow 해결

이건 단순 “HTML 앱” 수준을 넘어서,
이미 작은 프론트엔드 런타임/플랫폼 구조로 진입하고 있습니다.

그리고 당신이 말한:

> “티들리위키처럼 설정페이지 + 플러그인 모듈 + 유지보수 가능한 구조”

는 실제로 매우 중요한 다음 단계입니다.

다만 지금 시점에서는 기능 추가보다 먼저:

> “상태 관리 안정화 + lifecycle 정리 + 플러그인 경계 정의”

를 해야 합니다.

현재 코드베이스는 아직:

```text
global state + render side-effect + direct DOM mutation
```

패턴이 강하기 때문입니다.

이 상태에서 기능만 추가하면,
v0.7~v0.9쯤 가면 급격히 유지보수 난이도가 올라갈 가능성이 큽니다.

따라서 이후 작업은 아래 순서를 권장합니다.

# 1. 최우선 — Dirty State / 저장 보호 시스템

이건 반드시 빨리 넣어야 합니다.

현재 가장 위험한 부분입니다.

당신 구조는:

```text
local-first editor
```

인데,

브라우저 새로고침/탭 종료 시:

```text
미저장 상태 감지
```

가 아직 약합니다.

이건 실제 사용자 데이터 손실로 이어질 수 있습니다.

---

## 권장 구조

```js
S.meta.dirty = true
```

같은 dirty flag 시스템 도입.

그리고:

* 카드 수정
* 문서 수정
* 태그 수정
* 컬럼 변경
* 설정 변경

시 dirty=true.

저장 완료 시:

```js
dirty=false
```

---

## beforeunload 경고

```js
window.addEventListener('beforeunload', e => {
  if (!S.meta.dirty) return;
  e.preventDefault();
  e.returnValue = '';
});
```

---

## 더 중요한 부분

단순 경고보다 중요한 건:

# Auto Save Queue

입니다.

즉:

```text
수정 후 1~2초 debounce 저장
```

구조.

TiddlyWiki/Obsidian 계열 UX 핵심입니다.

---

# 2. 상태관리 계층 분리 (매우 중요)

현재 가장 큰 미래 리스크.

지금 구조는:

```text
render 함수 안에서 state 변경
```

이 일부 존재할 가능성이 높습니다.

이 패턴은 규모 커질수록 버그 폭증합니다.

---

## 추천 구조

### 현재

```text
UI → render() → 직접 save/render/sidebar 호출
```

### 권장

```text
UI
 → Action
   → State mutation
     → Render scheduler
```

즉:

```js
updateCard(id, patch)
```

같은 action layer 도입.

---

## 핵심 목표

앞으로는:

```js
save();
renderCards();
renderSidebar();
renderList();
```

를 여기저기 직접 호출하지 않게 해야 합니다.

대신:

```js
dispatch('CARD_UPDATED', payload)
```

같은 구조.

---

# 3. Render Scheduler 도입

이건 반드시 필요해질 겁니다.

현재 코드 패턴상:

```text
한 이벤트에서
renderCards()
renderSidebar()
renderList()
```

연속 호출 가능성이 높습니다.

그러면:

* 모바일 reflow 증가
* race condition
* flicker
* battery 소모

가 커집니다.

---

## 추천

```js
queueRender('cards');
queueRender('sidebar');
```

그리고:

```js
requestAnimationFrame(flushRenderQueue)
```

로 합치기.

# 왜 “완전 재개발”은 위험한가

소프트웨어 역사에서:

```text id="rn7znf"
“처음부터 다시 만들자”
```

는 매우 자주 실패했습니다.

대표적으로:

* Netscape → Mozilla 초창기
* Windows Longhorn
* 여러 노트앱 프로젝트들

공통 문제는:

```text id="zfq7sm"

```

한다는 점입니다.

그러면:

* 개발기간 폭증
* 완성도 저하
* 실제 사용 흐름 붕괴
* endless architecture redesign

로 갑니다.
