## 1. 요약

현재 `ol-atlas`는 **버전 0.0.7, schemaVersion 10**의 단일 HTML 기반 로컬 우선 문서 제작 도구다. 앱은 `src/index.html`, CSS, 번들 JS를 빌드 단계에서 하나의 HTML로 인라인하고, 사용자가 저장할 때는 현재 전체 state를 Base64로 인코딩해 HTML 내부의 `__LOADED_DATA_B64__` 상수에 삽입한다.

핵심 데이터 저장소는 서버가 아니라 브라우저 `localStorage`다. 파일 저장은 별도 백업/공유 포맷으로 동작하며, HTML 저장은 전체 프로젝트 복원용, JSON/Markdown 저장은 카드 교환용에 가깝다.

현재 빌드 검증 결과 `npm run build`는 성공했다. 산출물은 `dist/ol-atlas_v0.0.7.html`로 생성된다.

## 3. 스키마 상태

### 3.1 현재 스키마 버전

현재 목표 스키마는 `schemaVersion 10`이다. 기준 파일은 `src/core/schema.js`이며, 상수 `SCHEMA_CURRENT_VERSION = 10`으로 관리된다.

마이그레이션 체인은 다음과 같다.

| From | To | 주요 변경 |
|---:|---:|---|
| 6 | 7 | `meta.dirty`, `meta.lastSavedAt`, `settings` 기본값 추가 |
| 7 | 8 | `meta.editors`, `meta.saveLog`, `meta.currentEditorId`, `card.acts` 추가 |
| 8 | 9 | `meta.actLog` 추가 |
| 9 | 10 | `meta.bookInfo` 추가 |

현재 신규 state는 `src/core/state.js`의 `makeDefault()`에서 직접 `schemaVersion: 10`으로 생성된다.

### 3.2 현재 state 최상위 구조

현재 기본 state는 다음 구조를 갖는다.

```js
{
  meta: {
    fileId,
    title,
    created,
    version,
    schemaVersion,
    dirty,
    lastSavedAt,
    editors,
    saveLog,
    actLog,
    currentEditorId,
    bookInfo
  },
  settings,
  columns,
  cards,
  userData,
  nextColId,
  nextCardId,
  trash
}
```

### 3.3 meta 하위 주요 필드

- `fileId`: 파일 고유 식별자
- `title`: 파일 제목 및 저장 파일명 생성 기준
- `version`: 앱 빌드 버전에서 주입되는 문서 버전값
- `schemaVersion`: 현재 10
- `dirty`, `lastSavedAt`: autosave 상태 표시 및 이탈 경고 기준
- `editors`: 편집자 목록
- `saveLog`: HTML 파일 저장 시점의 편집자 저장 기록, 최대 200개
- `actLog`: 휴지통 영구삭제 시 카드별 `acts`를 보존하는 아카이브, 최대 1000개
- `currentEditorId`: 현재 세션 편집자 ID
- `bookInfo`: 책 제목, 부제, 저자, 역자, 출판사, 출판일, 개정일, 책 버전, 설명, 표지색, 언어, ISBN

### 3.4 카드 구조

카드는 현재 다음 성격의 필드를 사용한다.

- `id`, `colId`: 카드 ID와 컬럼 연결
- `title`, `body`: 제목과 마크다운 본문
- `group`, `tags`, `priority`, `created`
- `slug`: 문서뷰/카드 식별용 slug
- `images`: 본문 이미지 토큰 저장소
- `acts`: 편집자별 카드 생성/수정/삭제/복원 기록

이미지는 Markdown의 일반 이미지 문법을 내부적으로 `[img:id]` 토큰과 `card.images` 객체로 분리해 저장할 수 있다. Markdown 내보내기 시에는 다시 표준 이미지 문법으로 변환한다.

## 4. 저장 기능 상태

### 4.1 브라우저 로컬 저장

앱 내부 저장은 `localStorage` 기반이다.

- 현재 키: `ol_state`
- 레거시 키: `ol_v1`
- 저장 함수: `storageSave()`
- 불러오기 함수: `storageLoad()`

모든 action은 `dispatch()`를 통해 reducer를 거친 뒤 `markDirty()`를 호출한다. `markDirty()`는 1초 debounce 후 `storageSave()`를 실행하고 `markClean()`으로 `dirty=false`, `lastSavedAt=현재시각`을 기록한다.

즉, 일반 편집 변경은 “브라우저 로컬 자동저장”으로 보존된다. 다만 이 저장은 브라우저 프로필과 도메인/파일 실행 컨텍스트에 종속되므로, 사용자 간 공유나 장기 백업은 HTML 파일 저장 기능을 사용해야 한다.

### 4.2 OL HTML 파일 저장

상단 저장 버튼의 HTML 저장 흐름은 다음과 같다.

1. 현재 편집자를 확인한다.
2. 편집자가 없으면 편집자 모달을 띄운다.
3. `saveLog`에 `{ at, editorId }`를 추가한다.
4. `editors` 배열에 편집자를 upsert하고 저장 횟수를 갱신한다.
5. `updateEditors()`를 dispatch한다.
6. 현재 전체 state를 JSON 문자열로 만든다.
7. JSON을 Base64로 인코딩한다.
8. 현재 정적 HTML 템플릿의 `const __LOADED_DATA_B64__ = '...'` 값을 교체한다.
9. `파일제목_v문서버전.html`로 다운로드한다.

이 HTML 파일은 앱 코드와 데이터를 함께 포함하므로, 파일 하나만 전달해도 같은 데이터 상태를 복원할 수 있다.

중요한 점은 HTML 저장이 단순 export가 아니라 편집자 저장 이력을 갱신하는 행위라는 것이다. 따라서 저장 버튼을 누르면 `editors`, `saveLog`, `currentEditorId`가 함께 변한다.

### 4.3 JSON 저장

JSON 내보내기는 전체 state 백업이 아니라 카드 교환용 포맷이다.

포맷 식별자는 `ol-cards-v1`이며 구조는 다음과 같다.

```js
{
  format: "ol-cards-v1",
  exportedAt,
  source,
  cards: [...]
}
```

각 카드는 `id`, `title`, `body`, `column`, `group`, `tags`, `priority`, `learnStatus`, `created` 중심으로 저장된다. `meta`, `settings`, `bookInfo`, `editors`, `saveLog`, `actLog`, `trash`는 이 포맷에 포함되지 않는다.

### 4.4 Markdown 저장

Markdown 저장은 두 종류가 있다.

- 전체 카드를 하나의 `.md` 파일로 저장
- 각 카드를 개별 `.md` 파일로 만들고 zip으로 저장

각 카드 Markdown은 YAML 유사 frontmatter와 본문으로 구성된다.

```md
---
title: "..."
column: "..."
group: "..."
priority: mid
learnStatus: wait
tags: ["..."]
slug: "..."
created: ...
---

본문
```

개별 Markdown 내보내기에서는 `cardToMarkdownText()`가 사용되며, 이미지 토큰은 표준 Markdown 이미지 문법으로 복원된다. 단일 Markdown 내보내기 경로는 현재 `card.body`를 그대로 붙이는 방식이라, 이미지 토큰 복원 측면에서는 개별 Markdown 내보내기가 더 완성도가 높다.

## 5. 가져오기 기능 상태

### 5.1 OL HTML 파일 가져오기

HTML 가져오기는 전체 문서 복원 기능이다.

동작 흐름은 다음과 같다.

1. 사용자가 `.html` 파일을 선택한다.
2. 파일 텍스트에서 `const __LOADED_DATA_B64__ = '...'` 패턴을 찾는다.
3. 초기 플레이스홀더 `__INIT_DATA_B64__`이면 저장 데이터가 없는 파일로 판단한다.
4. Base64를 디코딩하고 JSON으로 파싱한다.
5. `columns`와 `cards`가 배열인지 최소 검증한다.
6. `normalizeState(data)`와 `migrate(...)`를 적용한다.
7. `applyState(...)`로 현재 앱 state를 교체한다.
8. `localStorage`에 저장한다.
9. 칸반 뷰로 전환하고 전체 렌더를 큐에 넣는다.

이 경로는 병합이 아니라 **현재 state 전체 교체**다. 기존 브라우저 로컬 데이터는 가져온 HTML 데이터로 덮인다.

### 5.2 JSON 가져오기

JSON 가져오기는 카드 병합 기능이다. 인식 가능한 입력은 두 가지다.

- 카드 배열 자체
- `{ format: "ol-cards-v1", cards: [...] }`

가져온 카드는 바로 적용되지 않고 병합 모달로 전달된다. 중복 감지는 다음 기준이다.

- `id`가 기존 카드와 일치
- 또는 `title + column` 조합이 기존 카드와 일치

사용 가능한 병합 전략은 다음과 같다.

- `skip`: 중복 카드 건너뛰기
- `overwrite`: 중복 카드 덮어쓰기
- `keepboth`: 중복이어도 새 카드로 추가

JSON 카드의 `column` 또는 `status` 이름이 현재 컬럼에 없으면 새 컬럼이 생성된다. `learnStatus`는 `wait`, `doing`, `done`만 반영된다.

### 5.3 Markdown 가져오기

Markdown 가져오기는 여러 `.md` 또는 `.markdown` 파일을 동시에 선택할 수 있다.

각 파일은 `parseMarkdownFile()`로 파싱된다.

- frontmatter가 있으면 `title`, `column`, `group`, `priority`, `learnStatus`, `tags`, `slug`, `created`를 읽는다.
- frontmatter가 없으면 첫 번째 `# 제목`을 제목으로 사용한다.
- 그래도 제목이 없으면 파일명을 fallback title로 사용한다.
- 본문 이미지 문법은 내부 이미지 토큰 구조로 변환된다.

파싱된 카드는 JSON 가져오기와 같은 병합 모달/전략을 사용한다.

## 6. 부팅과 데이터 우선순위

부팅 흐름은 `src/data/init.js`에 있다.

현재 순서는 다음과 같다.

1. `localStorage`에서 `ol_state`를 읽는다.
2. 없으면 레거시 `ol_v1`을 읽는다.
3. 저장 데이터가 없으면 `makeDefault()`로 기본 state를 만든다.
4. 저장 데이터가 있으면 migrate + normalize를 적용한다.
5. HTML 내부에 `__LOADED_DATA_B64__`가 있으면 그 데이터를 decode, migrate, normalize한 뒤 `localStorage`에 저장하고 state로 사용한다.
6. store를 초기화하고 전체 렌더 후 첫 화면을 home으로 전환한다.

중요한 점은 최종적으로 내장 HTML 데이터가 있으면 로컬 저장 데이터보다 우선한다는 것이다. 즉 저장된 OL HTML 파일을 열면 파일 안의 데이터가 브라우저 `ol_state`를 갱신한다.

## 7. 전문가에게 전달할 핵심 판단

### 7.1 현재 구현의 성격

OL ATLAS는 “서버 없는 단일 HTML 문서 앱”이다. 저장 기능은 두 층으로 나뉜다.

- 편집 중 자동저장: 브라우저 `localStorage`
- 공유/백업 저장: 전체 state를 품은 HTML 파일

JSON과 Markdown은 전체 문서 백업이 아니라 카드 단위 교환 포맷이다. 따라서 다른 전문가에게 기능을 설명할 때 “HTML은 전체 복원, JSON/Markdown은 카드 병합”으로 구분해야 한다.

### 7.2 스키마 안정성

스키마는 v6부터 v10까지의 마이그레이션 체인을 갖고 있어 과거 데이터 호환을 어느 정도 고려하고 있다. 현재 핵심 확장은 편집자/저장로그, 카드 행위 기록, 삭제 후 actLog 보존, 책 정보 bookInfo다.

다만 스키마 상수는 `schema.js`와 `build/inline.mjs`에 각각 존재한다. 현재는 둘 다 10으로 맞지만, 향후 스키마 변경 시 두 위치가 어긋날 수 있다. 가능하면 단일 상수 소스 또는 빌드 시 공유 모듈화가 필요하다.

### 7.3 저장/가져오기 리스크

- HTML 가져오기는 전체 state 교체 방식이므로 사용자에게 덮어쓰기 성격을 명확히 알려야 한다.
- HTML 가져오기 검증은 `columns`, `cards` 배열 여부 중심이라 엄격한 스키마 검증은 아니다.
- JSON/Markdown 병합의 중복 판정은 `id`와 `title+column` 기준이다. 제목이 같은 카드가 의도적으로 여러 개 있을 경우 사용자가 병합 전략을 잘 선택해야 한다.
- 단일 Markdown 내보내기는 이미지 토큰을 표준 Markdown 이미지로 복원하지 않는 경로가 있어, 이미지 포함 문서의 휴대성은 개별 Markdown zip 내보내기가 더 낫다.
- 저장 HTML은 현재 앱 코드까지 포함하므로, 오래된 HTML 파일은 오래된 앱 런타임을 함께 품는다. 최신 코드로 데이터를 가져오려면 최신 앱에서 HTML 가져오기를 수행하는 방식이 필요하다.

## 9. 현재 결론

현재 `ol-atlas`는 v0.0.7 기준으로 단일 HTML 저장/복원 모델이 정상적으로 자리 잡은 상태다. schemaVersion 10은 bookInfo까지 포함한 현재 운영 스키마이며, 과거 v6-v9 데이터에 대한 단계적 마이그레이션도 존재한다.

문서 저장과 가져오기 기능은 다음처럼 설명하는 것이 가장 정확하다.

- **HTML 저장**: 전체 state와 앱을 한 파일에 담는 완전 복원용 저장
- **HTML 가져오기**: 저장된 전체 state를 현재 앱에 덮어쓰는 복원
- **JSON 저장/가져오기**: 카드 목록 교환 및 병합
- **Markdown 저장/가져오기**: 카드 콘텐츠를 Markdown 파일로 교환 및 병합
- **localStorage 자동저장**: 편집 중 브라우저 로컬 보존

다음 단계에서 개선한다면, 우선순위는 스키마 상수 단일화, HTML 가져오기 전 덮어쓰기 경고 강화, 단일 Markdown 내보내기의 이미지 복원 일관화, 가져오기 payload 검증 강화 순서가 적절하다.

## 부록 A. OL ATLAS 전체 스키마 정보

이 부록은 `src/core/state.js`, `src/core/schema.js`, `src/core/normalize.js`, `src/actions/export-import.js`, `src/actions/card-actions.js` 기준으로 정리한 현재 OL ATLAS state 구조다. 문서상 설계가 아니라 현재 코드가 실제 저장·정규화·가져오기에서 사용하는 구조를 기준으로 한다.

### A.1 스키마 버전

- 현재 운영 스키마: `schemaVersion: 10`
- 현재 앱 버전: `package.json` 기준 `0.0.7`
- 빌드 메타: `window.__OL_BUILD__ = { version, schemaVersion, buildAt }`
- 저장 키: `localStorage["ol_state"]`
- 레거시 저장 키: `localStorage["ol_v1"]`

마이그레이션 체인:

| From | To | 추가/변경 필드 |
|---:|---:|---|
| 6 | 7 | `meta.dirty`, `meta.lastSavedAt`, `settings` |
| 7 | 8 | `meta.editors`, `meta.saveLog`, `meta.currentEditorId`, `card.acts` |
| 8 | 9 | `meta.actLog` |
| 9 | 10 | `meta.bookInfo` |


필드 설명:

| 필드 | 타입 | 기본값/예시 | 설명 |
|---|---|---|---|
| `fileId` | string | `ol-${random}` | 파일 단위 고유 ID |
| `title` | string | `OL Weaving the Wisdom` | 파일 제목, 저장 파일명 기준 |
| `created` | string | `YYYY-MM-DD` | 생성일 |
| `version` | string | 빌드 버전 또는 `0.0.1` | 파일/앱 버전 표시 |
| `schemaVersion` | number | `10` | 데이터 스키마 버전 |
| `dirty` | boolean | `false` | 미저장 변경 여부 |
| `lastSavedAt` | string/null | `null` | localStorage autosave 완료 시각 |
| `editors` | array | `[origin editor]` | 파일 저장에 참여한 편집자 목록 |
| `saveLog` | array | `[{ at, editorId, note }]` | HTML 파일 저장 이력, 최대 200개 |
| `actLog` | array | `[]` | 영구삭제된 카드의 행위 기록 아카이브, 최대 1000개 |
| `currentEditorId` | string/null | `null` | 현재 세션 편집자 |
| `bookInfo` | object | `BookInfo` | 책/문서집 메타데이터 |

기본 원본 편집자:


설명:

- HTML 파일 저장 버튼을 누를 때 추가된다.
- 기본 파일 생성 로그는 `{ at: now, editorId: "origin_biwoom", note: "원본 생성" }`이다.

```js
{

  editorId: string,
```

설명:

- `meta.currentEditorId`가 없으면 기록하지 않는다.

### A.10 `Column`

```js
{
  id: number | string,
  title: string,
  color: string,
  order?: number
}
```

기본 컬럼:

```js
[
  { id: 1, title: "😃 기획중", color: "rgb(99, 102, 241)" },
  { id: 2, title: "🤓 검토중", color: "rgb(245, 158, 11)" },
```

설명:

- 새 컬럼 ID는 `nextColId`를 사용한다.

### A.11 `Card`

```js
{
  id: number | string,
  colId: number | string | null,
  title: string,
  body: string,
  group: string,
  tags: string[],
  priority: "high" | "mid" | "low",
  created: string,
  slug: string,
  images: Record<string, CardImage>,
  acts: CardAct[]
}
```

필드 설명:

| 필드 | 타입 | 기본/허용값 | 설명 |
|---|---|---|---|
| `id` | number/string | `nextCardId` | 카드 ID |
| `colId` | number/string/null | 기존 컬럼 ID | 카드가 속한 컬럼 |
| `title` | string | `""` | 카드 제목 |
| `body` | string | `""` | Markdown 본문 |
| `group` | string | `""` | 카드 그룹명 |
| `tags` | string[] | `[]` | 카드 태그 |
| `priority` | string | `high`, `mid`, `low` | 우선순위. 유효하지 않으면 `mid` |
| `created` | string | `today()` | 생성일 |
| `slug` | string | title 기반 자동 생성 | 문서/URL 식별용 slug |
| `images` | object | `{}` | 본문 이미지 저장소 |
| `acts` | array | `[]` | 카드 행위 기록 |

정규화 규칙:

- `bodyMd`가 있고 `body`가 없으면 `body = bodyMd`로 옮긴 뒤 `bodyMd`는 삭제한다.
- `bodyMode`는 삭제한다.
- `history`는 삭제한다.
- `priority`가 `high | mid | low`가 아니면 `mid`로 바꾼다.
- `tags`가 배열이 아니면 빈 배열로 바꾼다.
- `images`가 객체가 아니면 빈 객체로 바꾼다.
- `slug`가 없으면 `titleToSlug(title)` 또는 카드 ID 기반으로 생성한다.
- 중복 slug가 있으면 `-2`, `-3` 형식으로 유일화한다.
- `acts`가 배열이 아니면 빈 배열로 바꾼다.

### A.12 `CardImage`

```js
{
  alt: string,
  src: string
}
```

본문 이미지 처리:

- 가져오기/정규화 중 Markdown 이미지 `![alt](src)`는 내부 토큰 `[img:img-1]` 형식으로 변환될 수 있다.
- 실제 이미지 데이터는 `card.images["img-1"] = { alt, src }`에 저장된다.
- 개별 Markdown 내보내기에서는 `[img:id]` 토큰을 다시 표준 Markdown 이미지 문법으로 복원한다.

### A.13 `userData`

```js
{
  status: {
    [cardId: string]: "wait" | "doing" | "done" | string
  }
}
```

설명:

- 카드별 학습/진행 상태를 저장한다.
- import/export에서 `learnStatus`로 표현된다.
- Markdown/JSON 가져오기에서는 `wait`, `doing`, `done`만 명시적으로 반영한다.
- 카드 삭제 시 해당 카드의 status는 삭제된다.

### A.14 `trash`

```js
TrashCard = Card & {
  _trashedAt: string
}
```

설명:

- 카드 삭제 시 `cards`에서 제거되고 `trash` 앞쪽에 추가된다.
- 삭제 시 `userData.status[cardId]`는 제거된다.
- 복원 시 기존 ID가 충돌하면 `nextCardId`로 새 ID를 부여한다.
- 영구삭제 시 해당 카드의 `acts`는 `meta.actLog`로 아카이브된다.

### A.15 ID 카운터

```js
nextColId: number
nextCardId: number
```

정규화 규칙:

- `nextColId`가 숫자가 아니거나 현재 최대 컬럼 ID보다 작거나 같으면 `maxColId + 1`로 보정한다.
- `nextCardId`가 숫자가 아니거나 현재 최대 카드 ID보다 작거나 같으면 `maxCardId + 1`로 보정한다.

### A.16 Markdown 가져오기 frontmatter 부분 스키마

현재 Markdown import가 읽는 frontmatter 키는 제한적이다.

```yaml
---
title: "..."
column: "..."
group: "..."
priority: high | mid | low
learnStatus: wait | doing | done
tags: ["..."]
slug: "..."
created: "YYYY-MM-DD"
---
```

파싱 결과:

```js
{
  title,
  column,
  group,
  priority,
  learnStatus,
  tags,
  created,
  body,
  slug,
  images: {}
}
```

주의:

- 위 목록 외 frontmatter 키는 현재 버려진다.
- 정식 YAML 파서가 아니라 단순 line parser다.
- 배열은 `tags: ["a", "b"]` 같은 inline 배열이 안전하다.
- 여러 줄 YAML 배열, 중첩 객체, 복잡한 문자열은 보존되지 않을 수 있다.

### A.17 Markdown 내보내기 부분 스키마

개별 카드 Markdown 내보내기 형식:

```yaml
---
title: "..."
column: "..."
group: "..."
priority: mid
learnStatus: wait
tags: ["..."]
slug: "..."
created: YYYY-MM-DD
---
```

단일 Markdown 내보내기 형식:

```yaml
---
title: "..."
column: "..."
group: "..."
priority: mid
learnStatus: wait
tags: ["..."]
created: YYYY-MM-DD
---
```

차이:

- 개별 Markdown 내보내기는 `slug`를 포함한다.
- 개별 Markdown 내보내기는 이미지 토큰을 표준 Markdown 이미지 문법으로 복원한다.
- 단일 Markdown 내보내기는 현재 `card.body`를 그대로 붙이므로 이미지 토큰 복원 일관성이 낮다.

### A.19 HTML 전체 저장 포맷

HTML 저장은 별도 JSON 파일이 아니라, 현재 전체 state를 다음 방식으로 HTML 안에 삽입한다.

```js
const __LOADED_DATA_B64__ = "Base64(JSON.stringify(state))";
```

복원 조건:

- HTML 파일 텍스트에서 `const __LOADED_DATA_B64__ = '...'` 패턴을 찾는다.
- 값이 `__INIT_DATA_B64__`이면 저장 데이터가 없는 기본 빌드 파일로 본다.
- 디코딩한 JSON에 `columns`와 `cards` 배열이 있어야 한다.
- 이후 `migrate()`와 `normalizeState()`를 적용한다.

### A.20 현재 스키마의 확장 여지

현재 Card에는 WORKS 같은 외부 원고의 전체 frontmatter를 보존할 전용 필드가 없다. WORKS 메타데이터를 손실 없이 가져오려면 다음 중 하나가 필요하다.

1. `card.sourceMeta` 또는 `card.frontmatter` 같은 객체 필드 추가
2. `prefixTags`를 `tags`에 병합하는 변환 규칙 사용
3. `series`, `seriesOrder`, `groupOrder`, `category`, `order`, `chapter`, `authors`, `license` 등을 ATLAS 표준 필드로 승격

현재 상태에서는 1번이 가장 안전하다. 기존 카드 표시 로직과 충돌하지 않고, OL HOME WORKS 원본 메타데이터를 보존할 수 있기 때문이다.
